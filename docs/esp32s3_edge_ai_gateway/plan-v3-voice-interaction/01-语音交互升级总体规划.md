# 语音交互升级总体规划（v3）

> 从"感知设备"到"交互设备"的本质跨越

---

## 版本对比

| 维度 | v1 (PIR+MLP) | v2 (多模态) | v3 (语音交互) |
|------|-------------|-------------|---------------|
| **核心能力** | 感知人体存在 | 感知+场景识别 | **感知+语音交互** |
| **用户交互** | 无 | 无 | **离线语音命令** |
| **技术框架** | Zephyr + TFLM | Zephyr + TFLM | **ESP-IDF + ESP-SR** |
| **AI 模型** | MLP (4KB) | 三模型 (35KB) | **WakeNet + MultiNet + TFLM** |
| **传感器** | PIR | 雷达+麦克风 | 雷达+麦克风+**DAC/扬声器** |
| **音频输出** | 无 | 无 | **TTS语音反馈** |
| **AEC回声消除** | 无 | 无 | **必须启用** |
| **决策模式** | 自动 | 自动 | **语音优先+自动兜底** |

## 升级动机

### 1. 用户体验痛点

| 痛点 | v2 解决方案 | v3 解决方案 |
|------|------------|------------|
| 无法手动干预 | 无 | "调亮一点" |
| 场景切换不直观 | 自动识别 | "打开演示模式" |
| 调试困难 | 串口命令 | "现在是什么模式" |
| 意外触发无法取消 | 无 | "恢复自动" |

### 2. 技术可行性

- **ESP-SR 成熟度**：Espressif 官方语音识别框架，生产级稳定性
- **ESP-IDF 生态**：完整的音频开发框架（ESP-ADF）
- **硬件兼容性**：ESP32-S3 内置 AI 加速指令，无需额外芯片

### 3. 架构演进

```
v1: PIR → MLP → 照明控制
v2: 雷达+麦克风 → 多模态融合 → 照明控制
v3: 雷达+麦克风 → 多模态融合 ─┬→ 照明控制
                              └→ 语音命令 → 优先级仲裁 → 照明控制
```

## AEC回声消除配置

### 为什么必须启用AEC？

```
┌─────────────────────────────────────────────────────────────────┐
│                    回声消除危机分析                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  无AEC时的灾难场景：                                             │
│                                                                 │
│  用户："小白网关"                                               │
│  设备：播放"好的，主人"（TTS）                                   │
│  麦克风：录入"好的，主人"                                        │
│  ESP-SR：识别到"好的，主人" → 可能触发其他命令                   │
│  结果：无限循环或误识别                                          │
│                                                                 │
│  有AEC时：                                                       │
│                                                                 │
│  ESP-ADF 从 DAC 输出端获取 Reference 信号                       │
│  AFE 算法将麦克风信号中的 Reference 成分消除                    │
│  麦克风：录入的是"纯净的用户语音"                               │
│  ESP-SR：正确识别或静默                                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### ESP-ADF AEC流水线配置

```c
// audio_pipeline.c 核心配置
audio_pipeline_handle_t setup_aec_pipeline(void) {
    // 1. 创建音频元素
    i2s_stream_cfg_t i2s_mic_cfg = I2S_STREAM_CFG_DEFAULT();
    i2s_mic_cfg.i2s_port = 0;  // 麦克风通道
    i2s_mic_cfg.type = AUDIO_STREAM_READER;

    i2s_stream_cfg_t i2s_dac_cfg = I2S_STREAM_CFG_DEFAULT();
    i2s_dac_cfg.i2s_port = 1;  // DAC通道
    i2s_dac_cfg.type = AUDIO_STREAM_WRITER;

    // 2. AEC处理器（关键！）
    afe_config_t afe_cfg = {
        .aec_filter = true,           // 启用回声消除
        .agc_mode = 3,                // 自动增益
        .noise_suppression = true,    // 噪声抑制
        .vad_init = true,             // 语音活动检测
        .reference_signal = I2S_PORT_1, // DAC输出作为参考信号
    };

    // 3. 构建流水线
    // 麦克风 → AEC → ESP-SR
    // DAC ← TTS ← 命令响应
}
```

## 麦克风流分时复用机制（隐患二解决方案）

### 问题：麦克风数据的"楚汉相争"

```
┌─────────────────────────────────────────────────────────────────┐
│                    资源抢占危机                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  v2 时代：麦克风 → MFCC → CNN → 场景分类（键盘声/会议声）        │
│  v3 时代：麦克风 → ESP-SR AFE → WakeNet/MultiNet（唤醒/命令）   │
│                                                                 │
│  冲突点：                                                        │
│  1. ESP-SR 的 AFE 需要独占 I2S 数据流进行降噪                    │
│  2. 同时拉数据给 TFLM 跑 CNN 会抽干 CPU 和 PSRAM 带宽            │
│  3. 结果：录音丢帧，语音识别失败                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案：VAD 动态路由

```c
// audio_router.c - 分时复用核心逻辑
typedef enum {
    ROUTE_TO_ESP_SR,    // 有人说话 → 语音识别优先
    ROUTE_TO_TFLM,      // 无人说话 → 场景分类
} audio_route_t;

static audio_route_t current_route = ROUTE_TO_TFLM;

void audio_router_task(void *arg) {
    while (1) {
        // ESP-SR AFE 内置 VAD 信号
        bool voice_detected = afe_get_vad_state();

        if (voice_detected && current_route == ROUTE_TO_TFLM) {
            // 有人说话，立即挂起 TFLM
            tflm_suspend();
            esp_sr_resume();
            current_route = ROUTE_TO_ESP_SR;
            ESP_LOGI(TAG, "切换到语音识别模式");
        }
        else if (!voice_detected && current_route == ROUTE_TO_ESP_SR) {
            // 无人说话，恢复 TFLM 场景分类
            esp_sr_suspend();
            tflm_resume();
            current_route = ROUTE_TO_TFLM;
            ESP_LOGI(TAG, "切换到场景分类模式");
        }

        vTaskDelay(pdMS_TO_TICKS(10));  // 10ms 检查周期
    }
}
```

### 任务优先级配置

| 任务 | 优先级 | 状态切换 | 说明 |
|------|--------|----------|------|
| ESP-SR 唤醒检测 | HIGH | VAD触发时激活 | 用户说话时必须响应 |
| TFLM 场景分类 | LOW | VAD触发时挂起 | 后台运行，可被打断 |
| 雷达数据处理 | MEDIUM | 始终运行 | 不受语音影响 |

## 仲裁器TTL租约机制（隐患三解决方案）

### 问题：僵化的优先级导致"退化为传统声控灯"

```
┌─────────────────────────────────────────────────────────────────┐
│                    灾难场景演示                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  下午 2 点：用户说"调暗一点"，系统调到 50%                       │
│  下午 6 点：天黑了，灯依然是 50%（被语音锁死）                   │
│  晚上 8 点：用户离开会议室，灯依然是 50%（死等"恢复自动"）       │
│                                                                 │
│  根本原因：语音命令没有"有效期"或"退出条件"                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案：带 TTL 的有限状态机

```c
// priority_arbiter.c - 核心状态机
typedef enum {
    MODE_AUTO,          // 自动模式：轨道一控制
    MODE_VOICE_LOCK,    // 语音锁定：用户干预生效
} arbiter_mode_t;

typedef struct {
    arbiter_mode_t mode;
    uint32_t voice_lock_ttl;        // 语音锁剩余时间（秒）
    uint32_t last_radar_change;     // 上次雷达状态变化时间
    bool radar_state_changed;       // 雷达状态是否发生重大变化
} arbiter_state_t;

void arbiter_update(arbiter_state_t *state, sensor_data_t *sensors) {
    if (state->mode == MODE_VOICE_LOCK) {
        // TTL 倒计时
        if (state->voice_lock_ttl > 0) {
            state->voice_lock_ttl--;
        }

        // 退出条件1：TTL 过期
        bool ttl_expired = (state->voice_lock_ttl == 0);

        // 退出条件2：雷达检测到环境重大变化
        // 例如：人离开超过 10 分钟再回来
        bool env_changed = state->radar_state_changed &&
                          (sensors->radar.presence_duration > 600);

        if (ttl_expired || env_changed) {
            state->mode = MODE_AUTO;
            ESP_LOGI(TAG, "语音锁释放，恢复自动控制");
            lighting_event_notify(EVENT_AUTO_RESUME);
        }
    }
}

// 语音命令触发时设置 TTL
void on_voice_command(voice_cmd_t cmd) {
    arbiter_state.mode = MODE_VOICE_LOCK;
    arbiter_state.voice_lock_ttl = 7200;  // 默认 2 小时
    arbiter_state.radar_state_changed = false;
}
```

| 任务 | 说明 | 优先级 |
|------|------|--------|
| ESP-IDF 项目初始化 | idf.py 创建项目骨架 | P0 |
| 目录结构搭建 | main/components/models | P0 |
| 构建系统配置 | CMakeLists.txt | P0 |
| 基础外设验证 | LED PWM、UART | P0 |

### Phase 2a: 纯语音控制LED（1 周）⭐ 快速验证核心风险

| 任务 | 说明 | 优先级 |
|------|------|--------|
| ESP-SR 组件集成 | 组件管理器添加 | P0 |
| WakeNet 唤醒词配置 | "小白网关" | P0 |
| MultiNet 命令词定义 | 基础亮度/色温命令 | P0 |
| 直接LED控制 | 跳过多模态，直接PWM | P0 |
| **端到端验证** | **语音命令 → LED响应** | **P0** |

> 此阶段完成后，可独立验证语音交互是否可行，降低后续风险。

### Phase 2b: 完整语音功能（1 周）

| 任务 | 说明 | 优先级 |
|------|------|--------|
| DAC + 扬声器集成 | MAX98357A 驱动 | P0 |
| AEC 回声消除配置 | ESP-ADF 流水线 | P0 |
| TTS 语音反馈 | 状态查询回复 | P1 |
| VAD 动态路由 | 麦克风流分时复用 | P0 |

### Phase 3: 多模态迁移（2 周）✅ 可与 Phase 2b 并行

| 任务 | 说明 | 优先级 |
|------|------|--------|
| TFLM 组件集成 | ESP-IDF 版本 | P0 |
| 模型文件迁移 | 三个 TFLite 模型 | P0 |
| 雷达驱动迁移 | Zephyr → ESP-IDF API | P0 |
| 推理逻辑迁移 | 多模态融合逻辑 | P0 |
| **统一事件格式** | **Lighting_Event_t** | **P0** |

### Phase 4: 仲裁机制（1 周）

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 优先级仲裁器 | 语音 > 自动 > 默认 | P0 |
| **TTL 租约状态机** | **语音锁自动释放** | **P0** |
| 统一队列消费 | xQueue → Lighting_Controller | P0 |

### Phase 5: 集成测试（1 周）

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 端到端测试 | 语音命令 → 照明响应 | P0 |

| 稳定性测试 | 长时间运行 | P1 |

## 语音命令设计

### 命令分类

| 类型 | 命令示例 | 数量 | 说明 |
|------|---------|------|------|
| **场景切换** | "打开专注模式" | 8 | 预设场景一键切换 |
| **亮度调节** | "调亮一点" | 6 | 相对/绝对调节 |
| **色温调节** | "换成暖光" | 6 | 色温控制 |
| **模式切换** | "切换演示模式" | 4 | 特殊模式 |
| **查询状态** | "现在是什么模式" | 5 | TTS 回复 |
| **取消操作** | "恢复自动" | 3 | 回到自动控制 |

### 命令词表（部分）

```c
// 场景切换
{"打开专注模式", CMD_SCENE_FOCUS,      {4000, 60}},
{"打开会议模式", CMD_SCENE_MEETING,    {5000, 80}},
{"打开演示模式", CMD_SCENE_PRESENTATION,{3000, 30}},
{"打开休息模式", CMD_SCENE_RELAX,       {2700, 40}},

// 亮度调节
{"调亮一点",    CMD_BRIGHT_UP,     {0, +20}},
{"调暗一点",    CMD_BRIGHT_DOWN,   {0, -20}},
{"亮度调到最大", CMD_BRIGHT_MAX,    {0, 100}},

// 色温调节
{"换成暖光",    CMD_COLOR_WARM,    {2700, 0}},
{"换成冷光",    CMD_COLOR_COOL,    {6500, 0}},

// 状态查询
{"现在是什么模式", CMD_QUERY_MODE,   {0, 0}},  // TTS 回复

// 取消操作
{"恢复自动",    CMD_AUTO_RESUME,   {0, 0}},  // 回到自动控制
```

## 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| ESP-SR 与 Zephyr 不兼容 | 高 | 高 | **已确认：迁移到 ESP-IDF** |
| 语音识别准确率低 | 中 | 中 | 增加命令词多样性、优化 AFE |
| 内存不足 | 中 | 高 | 分步加载、动态 Arena、启用PSRAM |
| 唤醒词误触发 | 中 | 中 | 调整阈值、增加确认机制 |
| 开发周期延长 | 中 | 中 | 分阶段交付、MVP 优先 |
| **AEC配置失败导致回声循环** | 高 | 高 | **严格按ESP-ADF示例配置Reference信号** |
| **DAC与麦克风I2S时钟冲突** | 中 | 高 | **使用独立I2S通道（I2S0/I2S1）** |
| **麦克风流资源抢占** | 高 | 高 | **VAD动态路由：无人声→TFLM，有人声→ESP-SR** |
| **语音锁死导致退化为传统声控灯** | 中 | 高 | **TTL租约机制：环境变化自动释放语音锁** |
| **大爆炸式迁移失败** | 中 | 高 | **Phase 2a先验证语音，并行开发多模态** |

## 成功指标

| 指标 | 目标值 | 验证方法 |
|------|--------|---------|
| 唤醒词识别率 | > 95% | 100 次唤醒测试 |
| 命令词识别率 | > 90% | 500 次命令测试 |
| 端到端延迟 | < 1s | 从唤醒到执行 |
| 内存占用 | < 1MB | 运行时峰值 |
| 固件大小 | < 2MB | Flash 占用 |
