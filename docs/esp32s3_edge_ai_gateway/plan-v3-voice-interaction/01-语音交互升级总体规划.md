## 升级动机

### 1. 用户体验痛点

| 痛点 | v2 解决方案 | v3 解决方案 |
|------|------------|------------|
| 无法手动干预 | 无 | "调亮一点" |
| 场景切换不直观 | 自动识别 | "打开演示模式" |
| 调试困难 | 串口命令 | "现在是什么模式" |
| 意外触发无法取消 | 无 | "恢复自动" |

### 2. 技术可行性

- **ESP-SR 成熟度**：Espressif 官方语音识别框架，生产级稳定性
- **ESP-IDF 生态**：完整的音频开发框架（ESP-ADF）
- **硬件兼容性**：ESP32-S3 内置 AI 加速指令，无需额外芯片

### 3. 架构演进

```
v1: PIR → MLP → 照明控制
v2: 雷达+麦克风 → 多模态融合 → 照明控制
v3: 雷达+麦克风 → 多模态融合 ─┬→ 照明控制
                              └→ 语音命令 → 优先级仲裁 → 照明控制
```

## AEC回声消除配置

### 为什么必须启用AEC？

```
┌─────────────────────────────────────────────────────────────────┐
│                    回声消除危机分析                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  无AEC时的灾难场景：                                             │
│                                                                 │
│  用户："小白小白"                                               │
│  设备：播放"好的，主人"（TTS）                                   │
│  麦克风：录入"好的，主人"                                        │
│  ESP-SR：识别到"好的，主人" → 可能触发其他命令                   │
│  结果：无限循环或误识别                                          │
│                                                                 │
│  有AEC时：                                                       │
│                                                                 │
│  ESP-ADF 从 DAC 输出端获取 Reference 信号                       │
│  AFE 算法将麦克风信号中的 Reference 成分消除                    │
│  麦克风：录入的是"纯净的用户语音"                               │
│  ESP-SR：正确识别或静默                                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### ESP-ADF AEC流水线配置

```c

audio_pipeline_handle_t setup_aec_pipeline(void) {

    i2s_stream_cfg_t i2s_mic_cfg = I2S_STREAM_CFG_DEFAULT();
    i2s_mic_cfg.i2s_port = 0;  // 麦克风通道

        .agc_mode = 3,                // 自动增益
        .noise_suppression = true,    // 噪声抑制
        .vad_init = true,             // 语音活动检测
        .reference_signal = I2S_PORT_1, // DAC输出作为参考信号
    };

    // 3. 构建流水线
    // 麦克风 → AEC → ESP-SR
    // DAC ← TTS ← 命令响应
}
```

## 麦克风流分时复用机制（隐患二解决方案）

### 问题：麦克风数据的"楚汉相争"

```
┌─────────────────────────────────────────────────────────────────┐
│                    资源抢占危机                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  v2 时代：麦克风 → MFCC → CNN → 场景分类（键盘声/会议声）        │
│  v3 时代：麦克风 → ESP-SR AFE → WakeNet/MultiNet（唤醒/命令）   │
│                                                                 │
│  冲突点：                                                        │
│  1. ESP-SR 的 AFE 需要独占 I2S 数据流进行降噪                    │
│  2. 同时拉数据给 TFLM 跑 CNN 会抽干 CPU 和 PSRAM 带宽            │
│  3. 结果：录音丢帧，语音识别失败                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案一：VAD 动态路由（轻量级）

```c
// audio_router.c - 分时复用核心逻辑
typedef enum {
    ROUTE_TO_ESP_SR,    // 有人说话 → 语音识别优先
    ROUTE_TO_TFLM,      // 无人说话 → 场景分类
} audio_route_t;

static audio_route_t current_route = ROUTE_TO_TFLM;

void audio_router_task(void *arg) {
    while (1) {
        // ESP-SR AFE 内置 VAD 信号
        bool voice_detected = afe_get_vad_state();

        if (voice_detected && current_route == ROUTE_TO_TFLM) {
            // 有人说话，立即挂起 TFLM
            tflm_suspend();
            esp_sr_resume();
            current_route = ROUTE_TO_ESP_SR;
            ESP_LOGI(TAG, "切换到语音识别模式");
        }
        else if (!voice_detected && current_route == ROUTE_TO_ESP_SR) {
            // 无人说话，恢复 TFLM 场景分类
            esp_sr_suspend();
            tflm_resume();
            current_route = ROUTE_TO_TFLM;
            ESP_LOGI(TAG, "切换到场景分类模式");
        }

        vTaskDelay(pdMS_TO_TICKS(10));  // 10ms 检查周期
    }
}
```

### 解决方案二：AFE独占 + RingBuffer消费者模型（推荐）⭐

> 当启用 AEC 回声消除时，ESP-SR AFE 变得极其沉重，必须 100% 独占 I2S DMA 读取权。

```
┌─────────────────────────────────────────────────────────────────┐
│                    AFE独占架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────┐         ┌─────────────────────────────────┐   │
│  │  I2S DMA    │ ──────▶ │  ESP-SR AFE (唯一生产者)         │   │
│  │  麦克风     │         │  - 降噪 (NS)                     │   │
│  │  INMP441    │         │  - 回声消除 (AEC)                │   │
│  └─────────────┘         │  - 自动增益 (AGC)                │   │
│                          │  - VAD 检测                      │   │
│                          └───────────────┬─────────────────┘   │
│                                          │                     │
│                                          ▼                     │
│                          ┌─────────────────────────────────┐   │
│                          │  FreeRTOS RingBuffer            │   │
│                          │  (干净的 PCM 音频数据)           │   │
│                          └───────────────┬─────────────────┘   │
│                                          │                     │
│                    ┌─────────────────────┴─────────────────┐   │
│                    ▼                                       ▼   │
│          ┌─────────────────┐                   ┌─────────────┐ │
│          │ 消费者A: TFLM   │                   │ 消费者B:     │ │
│          │ 场景分类 CNN    │                   │ MultiNet    │ │
│          │ (后台运行)      │                   │ 命令识别    │ │
│          └─────────────────┘                   └─────────────┘ │
│                                                                 │
│  ⚠️ 关键：TFLM 和 MultiNet 从 RingBuffer 取数据，              │
│     而不是去底层 I2S 驱动抢数据                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```c
// audio_afe.c - AFE独占架构核心实现
#include "freertos/ringbuf.h"

static RingbufHandle_t audio_ringbuf = NULL;

#define AUDIO_RINGBUF_SIZE  (8 * 1024)  // 8KB 环形缓冲

void audio_pipeline_init(void) {
    // 1. 创建 RingBuffer
    audio_ringbuf = xRingbufferCreate(AUDIO_RINGBUF_SIZE, RINGBUF_TYPE_NOSPLIT);

    // 2. AFE 作为唯一生产者
    afe_config_t afe_cfg = AFE_CONFIG_DEFAULT();
    afe_cfg.aec_filter = true;
    afe_cfg.vad_init = true;
    afe_cfg.agc_mode = 3;

    esp_afe_sr_models_t *models = esp_afe_sr_models_init();
    esp_afe_sr_data_t *afe_data = esp_afe_sr_create(&afe_cfg, models);
}

// AFE 生产者任务：从 I2S 读取 → AFE 处理 → 写入 RingBuffer
void afe_producer_task(void *arg) {
    int audio_chunk_size = 512;  // 每次处理 512 samples
    int16_t *audio_buffer = malloc(audio_chunk_size * sizeof(int16_t));

    while (1) {
        // 1. 从 I2S DMA 读取原始音频（AFE独占）
        i2s_read(I2S_NUM_0, audio_buffer, audio_chunk_size * 2, portMAX_DELAY);

        // 2. AFE 处理（降噪、AEC、AGC）
        int16_t *clean_audio = esp_afe_sr_process(afe_data, audio_buffer);

        // 3. 写入 RingBuffer（消费者各取所需）
        xRingbufferSend(audio_ringbuf, clean_audio, audio_chunk_size * 2, pdMS_TO_TICKS(100));
    }
}

// TFLM 消费者：从 RingBuffer 取数据进行场景分类
void tflm_consumer_task(void *arg) {
    size_t item_size;
    int16_t *audio_data;

    while (1) {
        audio_data = (int16_t *)xRingbufferReceive(audio_ringbuf, &item_size, pdMS_TO_TICKS(100));
        if (audio_data) {
            // 运行 MFCC + CNN 场景分类
            run_scene_classification(audio_data, item_size / 2);
            vRingbufferReturnItem(audio_ringbuf, audio_data);
        }
    }
}
```

### 任务优先级配置

| 任务 | 优先级 | 状态切换 | 说明 |
|------|--------|----------|------|
| AFE 生产者 | HIGH | 始终运行 | 独占 I2S，保证音频不丢帧 |
| ESP-SR 唤醒检测 | HIGH | VAD触发时激活 | 用户说话时必须响应 |
| TFLM 场景分类 | LOW | 后台消费 | 从 RingBuffer 取数据 |
| 雷达数据处理 | MEDIUM | 始终运行 | 不受语音影响 |

## 仲裁器TTL租约机制（隐患三解决方案）

### 问题：僵化的优先级导致"退化为传统声控灯"

```
┌─────────────────────────────────────────────────────────────────┐
│                    灾难场景演示                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  下午 2 点：用户说"调暗一点"，系统调到 50%                       │
│  下午 6 点：天黑了，灯依然是 50%（被语音锁死）                   │
│  晚上 8 点：用户离开会议室，灯依然是 50%（死等"恢复自动"）       │
│                                                                 │
│  根本原因：语音命令没有"有效期"或"退出条件"                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案：带 TTL 的有限状态机

```c
// priority_arbiter.c - 核心状态机
typedef enum {
    MODE_AUTO,          // 自动模式：轨道一控制
    MODE_VOICE_LOCK,    // 语音锁定：用户干预生效
} arbiter_mode_t;

typedef struct {
    arbiter_mode_t mode;
    uint32_t voice_lock_ttl;        // 语音锁剩余时间（秒）
    uint32_t last_radar_change;     // 上次雷达状态变化时间
    bool radar_state_changed;       // 雷达状态是否发生重大变化
} arbiter_state_t;

void arbiter_update(arbiter_state_t *state, sensor_data_t *sensors) {
    if (state->mode == MODE_VOICE_LOCK) {
        // TTL 倒计时
        if (state->voice_lock_ttl > 0) {
            state->voice_lock_ttl--;
        }

        // 退出条件1：TTL 过期
        bool ttl_expired = (state->voice_lock_ttl == 0);

        // 退出条件2：雷达检测到环境重大变化
        // 例如：人离开超过 10 分钟再回来
        bool env_changed = state->radar_state_changed &&
                          (sensors->radar.presence_duration > 600);

        if (ttl_expired || env_changed) {
            state->mode = MODE_AUTO;
            ESP_LOGI(TAG, "语音锁释放，恢复自动控制");
            lighting_event_notify(EVENT_AUTO_RESUME);
        }
    }
}

// 语音命令触发时设置 TTL
void on_voice_command(voice_cmd_t cmd) {
    arbiter_state.mode = MODE_VOICE_LOCK;
    arbiter_state.voice_lock_ttl = 7200;  // 默认 2 小时
    arbiter_state.radar_state_changed = false;
}
```

## 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| ESP-SR 与 Zephyr 不兼容 | 高 | 高 | **已确认：迁移到 ESP-IDF** |
| 语音识别准确率低 | 中 | 中 | 增加命令词多样性、优化 AFE |
| 内存不足 | 中 | 高 | 分步加载、动态 Arena、启用PSRAM |
| 唤醒词误触发 | 中 | 中 | 调整阈值、增加确认机制 |
| 开发周期延长 | 中 | 中 | 分阶段交付、MVP 优先 |
| **AEC配置失败导致回声循环** | 高 | 高 | **严格按ESP-ADF示例配置Reference信号** |
| **DAC与麦克风I2S时钟冲突** | 中 | 高 | **使用独立I2S通道（I2S0/I2S1）** |
| **麦克风流资源抢占** | 高 | 高 | **VAD动态路由：无人声→TFLM，有人声→ESP-SR** |
| **语音锁死导致退化为传统声控灯** | 中 | 高 | **TTL租约机制：环境变化自动释放语音锁** |
| **大爆炸式迁移失败** | 中 | 高 | **Phase 2a先验证语音，并行开发多模态** |
