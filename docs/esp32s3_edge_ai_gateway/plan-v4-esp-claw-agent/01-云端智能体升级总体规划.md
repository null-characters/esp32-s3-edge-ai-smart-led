# 云端智能体升级总体规划（v4）

> 从"交互设备"到"主动智能体（Agent）"的本质跨越

---

## 1. 项目背景与目标

### 1.1 当前状态（v3）

v3 版本已实现完整的离线多模态交互能力：

| 能力 | 实现状态 | 技术方案 |
|------|---------|---------|
| 雷达感知 | ✅ 已实现 | ESP32-S3 ADC + FFT |
| 语音唤醒 | ✅ 已实现 | ESP-SR 唤醒词 |
| 语音识别 | ✅ 已实现 | 本地命令词识别 |
| 语音合成 | ✅ 已实现 | ESP-TTS 离线合成 |
| 多模态融合 | ✅ 已实现 | 手势+语音+雷达 |
| 智能控制 | ✅ 已实现 | 本地 MLP 模型推理 |

### 1.2 核心痛点

| 痛点 | 描述 | 影响 |
|------|------|------|
| **交互僵化** | 只能识别固定命令词，无法理解自然语言 | 用户体验差 |
| **规则固定** | C 语言硬编码逻辑，无法动态更新 | 迭代成本高 |
| **无记忆** | 每次交互独立，无法学习用户习惯 | 智能化程度低 |
| **无主动能力** | 只能被动响应，无法主动服务 | 设备价值有限 |

### 1.3 v4 目标

**核心定位**：从"交互设备"升级为"主动智能体"

| 目标 | 描述 | 衡量指标 |
|------|------|---------|
| 自然语言理解 | 支持自然语言指令，无需固定命令词 | 识别准确率 >90% |
| 动态规则 | 云端下发 Lua 脚本，本地执行 | 规则热更新 <5s |
| 五维记忆 | Profile/Preference/Fact/Rule/Context | 记忆召回准确率 >85% |
| 主动服务 | 基于历史行为预测用户需求 | 预测准确率 >70% |
| 优雅降级 | 网络断开时回退 v3 本地模式 | 降级时间 <1s |

## 2. 版本对比

| 维度 | v3 (离线完全态) | v4 (云端智能体) |
|------|----------------|-----------------|
| **核心能力** | 感知+语音交互 | **感知+云端Agent** |
| **响应速度** | 极快 (<200ms) | L1本地快 + L2云端慢 |
| **理解力** | 固定命令词 | **自然语言理解** |
| **灵活性** | 逻辑固定 (C 语言) | **动态脚本生成 (Lua)** |
| **记忆能力** | 无 | **五维记忆系统** |
| **主动能力** | 无 | **预测性服务** |
| **网络依赖** | 完全离线 | 混合模式（可降级） |

### 4.2 工具调用（Tool Calling）

| 工具名称 | 功能描述 | 参数 | 返回值 |
|---------|---------|------|--------|
| `set_lighting` | 调节灯光亮度和色温 | `brightness`: 0-100<br>`color_temp`: 2700-6500K | `success`: bool |
| `get_sensor_data` | 获取传感器数据 | 无 | `radar_distance`: float<br>`ambient_lux`: float<br>`gesture`: string |
| `send_notify` | 发送通知消息 | `message`: string<br>`channel`: "telegram"\|"feishu" | `success`: bool |
| `set_scene` | 切换场景模式 | `scene_name`: string | `success`: bool |
| `get_user_preference` | 获取用户偏好 | `user_id`: string<br>`preference_key`: string | `value`: any |
| `update_memory` | 更新记忆 | `memory_type`: string<br>`key`: string<br>`value`: any | `success`: bool |

### 6.2 Lua 脚本执行环境

```lua
-- Lua 脚本沙箱环境
-- 可用 API:
--   - set_lighting(brightness, color_temp)
--   - get_sensor_data() -> {radar, lux, gesture}
--   - get_user_preference(key) -> value
--   - log(message)
--   - send_notify(message, channel)

-- 示例: 智能光照补偿规则
function auto_lighting_rule()
    local sensor = get_sensor_data()
    local user_pref = get_user_preference("brightness_sensitivity")
    
    -- 根据环境光照自动调节
    if sensor.lux > 600 then
        -- 环境亮，降低灯光
        set_lighting(50 + user_pref.offset, 4500)
    elseif sensor.lux < 200 then
        -- 环境暗，提高灯光
        set_lighting(90 + user_pref.offset, 3500)
    end
    
    -- 有人靠近时提高亮度
    if sensor.radar < 100 then
        set_lighting(100, 4000)
    end
end

-- 注册为定时回调
register_callback("on_sensor_update", auto_lighting_rule)
```

## 10. 关键技术风险与规避

### 10.1 风险一：大模型长文本与本地 TTS 算力雪崩

#### 问题描述

| 维度 | v3 本地模式 | v4 云端模式 |
|------|------------|------------|
| TTS 文本 | 固定短语（"已为您开灯"） | 动态长文本（大模型生成） |
| 文本长度 | <20 字符 | 可能 >100 字符 |
| 算力需求 | 低，可本地实时合成 | 高，本地无法承受 |

**致命场景**：大模型可能返回冗长的回复，例如：
> "主人，我已经根据您昨晚的习惯，将灯光调至 4000K 专注模式，检测到您现在心率略高，建议调暗灯光放松一下，需要帮您向飞书发送加班状态吗？"

ESP32-S3 无法在本地实时将这么长的动态文本合成高质量语音，强行运行会导致雷达检测等其他任务严重卡顿。

#### 解决方案：Cloud Streaming TTS

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      云端流式 TTS 架构                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  云端分支流程:                                                               │
│                                                                             │
│  用户语音 ──► 云端 STT ──► 大模型生成文本 ──► 云端 TTS API                    │
│                                                         │                   │
│                                                         ▼                   │
│                                              ┌─────────────────────┐        │
│                                              │ 音频流 (MP3/Opus)   │        │
│                                              └──────────┬──────────┘        │
│                                                         │                   │
│                                                         ▼                   │
│  ESP32-S3:                               ┌─────────────────────┐            │
│                                          │ HTTP Stream 接收    │            │
│                                          │ ESP-ADF 解码器      │            │
│                                          │ I2S 边收边播        │            │
│                                          └─────────────────────┘            │
│                                                                             │
│  本地分支流程（降级模式）:                                                   │
│                                                                             │
│  本地指令 ──► 本地 TTS（固定短语）──► I2S 播放                               │
│                                                                             │
│  策略:                                                                      │
│  - 云端模式: 使用云端 TTS + HTTP Stream 流式播放                            │
│  - 降级模式: 回退本地 TTS，播报简短固定反馈                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 技术实现要点

| 要点 | 实现方式 |
|------|---------|
| 云端 TTS 服务 | 阿里云 TTS / 微软 Azure TTS / OpenAI TTS API |
| 音频格式 | MP3 或 Opus（压缩率高，带宽占用低） |
| 流式播放 | ESP-ADF 的 `audio_element_stream` 组件 |
| 缓冲策略 | 环形缓冲区，缓冲 2-3 秒后开始播放 |
| 降级切换 | 网络断开时自动切换本地 TTS |

### 10.2 风险二：事件风暴导致的 Token 破产

#### 问题描述

| 场景 | 事件频率 | 日事件量 | Token 消耗估算 |
|------|---------|---------|---------------|
| 雷达微动检测 | 1次/秒 | 86,400 | ~$50-100/天 |
| 传感器状态变化 | 不定 | 不定 | 不可控 |

**致命场景**：LD2410 雷达极其灵敏，每秒检测到微动就生成 Event 上报云端，会导致：
1. Wi-Fi 芯片持续高负载，过热风险
2. API Token 费用爆炸式增长
3. 云端大模型被无效请求淹没

#### 解决方案：事件节流与边缘聚合

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      事件清洗层架构                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  传感器数据流:                                                               │
│                                                                             │
│  ┌─────────┐    ┌─────────────────────────────────────────────────────┐    │
│  │ LD2410  │───►│              事件清洗层 (Event Filter)               │    │
│  │ 雷达    │    │                                                     │    │
│  └─────────┘    │  规则一: 状态突变上报                                  │    │
│                 │  ─────────────────────────────────────────────────  │    │
│  ┌─────────┐    │  只有状态发生根本性翻转时才上报:                        │    │
│  │ 环境光  │───►│  - "无人" → "有人" (且持续 3 分钟确认)                │    │
│  │ 传感器  │    │  - "开会" → "静坐"                                    │    │
│  └─────────┘    │  - 亮度变化 > 20%                                     │    │
│                 │                                                     │    │
│                 │  规则二: 伴随式上报                                    │    │
│                 │  ─────────────────────────────────────────────────  │    │
│                 │  平时不上报，只有用户唤醒时才打包当前状态:               │    │
```

### 10.3 风险三：Lua 虚拟机的脱缰野马（大模型幻觉）

#### 问题描述

**致命场景**：大模型可能生成有问题的 Lua 代码：

```lua
-- 场景一: 死循环
while true do
    set_lighting(100)  -- CPU 100% 占用
end

-- 场景二: 高频硬件调用
for i = 1, 10000 do
    set_lighting(i % 100)  -- 100Hz 开关 LED，可能烧毁驱动
end

-- 场景三: 危险系统调用
os.execute("rm -rf /")  -- 如果未屏蔽，后果严重
```

一旦下发到 ESP32：
- 看门狗检测到死锁，无限重启
- 高频硬件调用可能烧毁 LED 驱动电路
- 系统完全失控

#### 解决方案：沙盒与硬件调用限流


### 10.4 风险四：原音频直接上传的带宽黑洞

#### 问题描述

| 音频格式 | 参数 | 数据量（5秒） | 上传延迟估算 |
|---------|------|-------------|-------------|
| PCM 原始 | 16kHz, 16bit, 单声道 | 160KB | 3-5秒 |
| OPUS 压缩 | 16kHz, 16kbps | 10KB | <0.5秒 |
| AMR-WB | 16kHz, 23.85kbps | 15KB | <0.5秒 |

**致命场景**：直接上传 I2S 采集的 PCM 原始音频：
- 一秒钟数据量：16kHz × 16bit × 1ch = 32KB
- 5 秒指令：160KB 原始数据
- 在 IoT 拥挤 Wi-Fi 环境下，上传延迟可达 3-5 秒
- 用户体验极差："说完话要等好几秒才有反应"

#### 解决方案：端侧音频压缩

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      音频处理流水线                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐      │
│  │ I2S     │    │             │    │ ESP-ADF     │    │ HTTPS       │      │
│  │ ADC     │───►│ PCM 缓冲    │───►│ Encoder     │───►│ POST        │      │
│  │ 16kHz   │    │             │    │ (OPUS/AMR)  │    │ 上传        │      │
│  └─────────┘    └─────────────┘    └─────────────┘    └─────────────┘      │
│       │                                                   │                │
│       │              数据量变化                            │                │
│       │              ────────────                          │                │
│       │              PCM: 160KB (5秒)                     │                │
│       │                    │                              │                │
│       │                    ▼                              │                │
│       │              OPUS: 10KB (压缩比 16:1)              │                │
│       │                                                     │                │
│       ▼                                                     ▼                │
│  ┌─────────────────────────────────────────────────────────────────────┐  │
│  │                         端到端延迟分析                                │  │
│  │                                                                     │  │
│  │  PCM 直传:  录音(5s) + 上传(3-5s) + STT(1s) + LLM(2s) + TTS(1s)     │  │
│  │            = 12-14 秒                                                │  │
│  │                                                                     │  │
│  │  OPUS压缩: 录音(5s) + 编码(0.2s) + 上传(0.3s) + STT(1s) + LLM(2s)   │  │
│  │            + TTS流式(0s等待) = 8.5 秒                               │  │
│  │                                                                     │  │
│  │  节省: 3.5-5.5 秒 (约 30% 延迟削减)                                  │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 11. 技术难点与解决方案

| 难点 | 风险等级 | 解决方案 |
|------|---------|---------|
| **内存压力** | 高 | 冷热数据分离、内存池复用、Lua 脚本大小限制 |

| **调试困难** | 低 | 远程日志、状态上报、IM 通知 |
