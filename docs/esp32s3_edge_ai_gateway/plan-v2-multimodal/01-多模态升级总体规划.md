## 一、升级动机与价值分析

### 1.1 当前方案的局限性

| 维度 | 当前状态 | 问题 |
|------|----------|------|
| **传感器** | PIR (0/1) | 信息熵 ≈ 1 bit，一个 `if` 就能处理 |
| **模型** | MLP 7→16→2 | 本质是线性拟合，if-else 可替代 |
| **AI 价值** | 几乎为零 | 没有解决"C语言写不出来"的问题 |
| **资源利用** | 模型 2.6KB | PSRAM 8MB 利用率 < 0.1% |

### 1.2 升级后的价值对比

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    传感器信息熵对比                                       │
├──────────────────┬────────────────────┬──────────────────────────────────┤
│ 传感器            │ 信息熵             │ AI 处理必要性                     │
├──────────────────┼────────────────────┼──────────────────────────────────┤
│ PIR              │ ~1 bit             │ ✗ 无需 AI，if-else 足够           │
│ 毫米波雷达        │ ~数百 bit/帧       │ ✓ 必须 AI，波形特征提取            │
│ MEMS 麦克风       │ ~数千 bit/帧       │ ✓ 必须 AI，频谱模式识别            │
│ 雷达 + 麦克风     │ ~数千 bit/帧       │ ✓ 多模态融合，C 语言无法实现        │
└──────────────────┴────────────────────┴──────────────────────────────────┘
```

### 1.3 AI 的真正价值：解决"C语言写不出来"的问题

| 场景 | PIR | 雷达 | 麦克风 | AI 能区分？ |
|------|-----|------|--------|-------------|
| 人静坐呼吸 | 1 | 呼吸微动特征 | 寂静 | ✓ 呼吸频段 0.1-0.5Hz |
| 风扇摇头 | 1 | 周期运动 | 低频嗡嗡 | ✓ 运动模式 + 声音特征 |
| 窗帘飘动 | 1 | 低能量抖动 | 无 | ✓ 能量阈值过滤 |
| 会议高潮 | 1 | 多人微动 | 多人声嘈杂 | ✓ 声音能量 + 人声检测 |
| 专注工作 | 1 | 单人静坐 | 键盘声 | ✓ 声音模式识别 |
| 打扫卫生 | 1 | 不规则移动 | 偶发声音 | ✓ 运动轨迹分析 |

## 二、硬件选型方案

### 2.1 毫米波雷达选型

| 型号 | 接口 | 价格 | 特点 | 推荐度 |
|------|------|------|------|--------|
| **LD2410** | UART | ~¥50 | 24GHz，支持多目标，成熟稳定 | ⭐⭐⭐⭐⭐ |
| LD2410C | UART | ~¥60 | LD2410 升级版，更灵敏 | ⭐⭐⭐⭐ |
| RCWL-0516 | GPIO | ~¥10 | 便宜但只有存在检测，无距离/速度 | ⭐⭐ |
| Aqara FP1 | Zigbee | ~¥200 | 成品方案，集成度高 | ⭐⭐⭐ |

**推荐：LD2410**
- UART 接口，ESP32-S3 双 UART，不影响调试
- 输出数据：距离(0-8m)、速度、能量、目标数量
- 功耗低，适合常开

### 2.2 MEMS 麦克风选型

| 型号 | 接口 | 价格 | 特点 | 推荐度 |
|------|------|------|------|--------|
| **INMP441** | I2S | ~¥15 | 高质量，I2S 直连 ESP32 | ⭐⭐⭐⭐⭐ |
| MSM261S4030H0 | I2S | ~¥10 | 便宜，性能略低 | ⭐⭐⭐⭐ |
| SPH0645LU4H | I2S | ~¥20 | 带 PDM 输出，低噪声 | ⭐⭐⭐⭐ |

**推荐：INMP441**
- I2S 接口，ESP32-S3 硬件 I2S 支持
- 24-bit，高质量音频采集
- 可配合 I2S 放大器做本地回放（调试用）

### 3.2 声音分类器（模型 A）

**任务**：将环境音分类为 5 种场景

| 类别 | 描述 | 特征 |
|------|------|------|
| 0 | 寂静 | 能量极低 |
| 1 | 单人讲话 | 人声频段 300-3400Hz |
| 2 | 多人讨论 | 多人声，能量高 |
| 3 | 键盘敲击 | 高频脉冲，周期性 |
| 4 | 其他噪音 | 风扇、空调等 |

**模型结构**：
```python
sound_classifier = tf.keras.Sequential([
    # 输入: MFCC (40, 32, 1) - 40 个 MFCC 系数，32 帧
    tf.keras.layers.Conv2D(16, (3, 3), activation='relu', input_shape=(40, 32, 1)),
    tf.keras.layers.MaxPooling2D((2, 2)),
    tf.keras.layers.Conv2D(32, (3, 3), activation='relu'),
    tf.keras.layers.MaxPooling2D((2, 2)),
    tf.keras.layers.Flatten(),
    tf.keras.layers.Dense(32, activation='relu'),
    tf.keras.layers.Dense(5, activation='softmax')  # 5 类场景
])
# 模型大小: ~30KB (INT8 量化后)
```

### 3.3 雷达分析器（模型 B）

**任务**：从雷达数据提取运动特征

**输入特征**（来自 LD2410）：
```c
typedef struct {
    float distance;          // 目标距离 0-8m
    float velocity;          // 目标速度 -5~+5 m/s
    float energy;            // 目标能量/信噪比
    float target_count;      // 目标数量
    float motion_amplitude;  // 运动幅度（自定义特征）
    float motion_period;     // 运动周期（检测呼吸/风扇）
    float distance_variance; // 距离方差（多人 vs 单人）
    float energy_trend;      // 能量趋势（进入/离开）
} radar_features_t;  // 8 维
```

**模型结构**：
```python
radar_analyzer = tf.keras.Sequential([
    tf.keras.layers.Dense(16, activation='relu', input_shape=(8,)),
    tf.keras.layers.Dense(12, activation='relu'),
    tf.keras.layers.Dense(6, activation='linear')  # 输出: 6 维特征
])
# 模型大小: ~2KB
```

### 3.4 多模态融合层（模型 C）

**任务**：融合声音、雷达、环境特征，输出最终控制决策

**输入**：
- 声音分类结果：5 维（one-hot 或 softmax 概率）
- 雷达分析结果：6 维
- 环境特征：8 维（时间、天气、日落等）

**模型结构**：
```python
fusion_model = tf.keras.Sequential([
    tf.keras.layers.Dense(32, activation='relu', input_shape=(19,)),
    tf.keras.layers.Dense(16, activation='relu'),
    tf.keras.layers.Dense(2, activation='sigmoid')  # 色温、亮度
])
# 模型大小: ~3KB
```

### 3.5 总模型大小估算

| 模型 | Float32 | INT8 量化后 |
|------|---------|-------------|
| 声音分类器 | ~120 KB | ~30 KB |
| 雷达分析器 | ~8 KB | ~2 KB |
| 融合决策器 | ~12 KB | ~3 KB |
| **总计** | ~140 KB | ~35 KB |

**PSRAM 利用率**：35KB / 8MB ≈ 0.4%，仍有大量空间用于：
- 音频缓冲区（16kHz × 2s = 32KB）
- 雷达历史数据（10Hz × 60s × 8 维 = 4.8KB）
- 推理中间结果

```c
#define SAMPLE_RATE     16000   // 16kHz

#define BUFFER_COUNT    4       // 4 个 DMA 缓冲区
#define BUFFER_SIZE     1024    // 每个缓冲区 1024 采样点

static const struct device *i2s_dev = DEVICE_DT_GET(DT_NODELABEL(i2s0));

int inmp441_init(void)
{
    struct i2s_config i2s_cfg = {
        .word_size = SAMPLE_BITS,
        .channels = 1,  // 单声道
        .format = I2S_FMT_DATA_FORMAT_I2S,
        .options = I2S_OPT_BIT_CLK_MASTER | I2S_OPT_FRAME_CLK_MASTER,
        .frame_clk_freq = SAMPLE_RATE,
        .block_size = BUFFER_SIZE * sizeof(int16_t),
        .timeout = 1000,
    };
    
    return i2s_configure(i2s_dev, I2S_DIR_RX, &i2s_cfg);
}
```

### 5.4 多模型推理调度

```c
// prj/src/logic/multimodal_infer.c

typedef struct {
    float sound_probs[5];     // 声音分类概率
    float radar_features[6];  // 雷达特征
    float env_features[8];    // 环境特征
} multimodal_input_t;

typedef struct {
    uint16_t color_temp;
    uint8_t brightness;
    uint8_t scene_id;         // 识别的场景 ID
} multimodal_output_t;

/**
 * @brief 多模态推理主函数
 */
int multimodal_infer(multimodal_input_t *input, multimodal_output_t *output)
{
    // 1. 声音分类器推理
    float sound_feat[5];
    tflm_invoke(sound_model, input->sound_probs, sound_feat);
    
    // 2. 雷达分析器推理
    float radar_feat[6];
    tflm_invoke(radar_model, input->radar_features, radar_feat);
    
    // 3. 拼接融合层输入
    float fusion_input[19];
    memcpy(fusion_input, sound_feat, 5 * sizeof(float));
    memcpy(fusion_input + 5, radar_feat, 6 * sizeof(float));
    memcpy(fusion_input + 11, input->env_features, 8 * sizeof(float));
    
    // 4. 融合层推理
    float fusion_out[2];
    tflm_invoke(fusion_model, fusion_input, fusion_out);
    
    // 5. 后处理
    output->color_temp = (uint16_t)(fusion_out[0] * 3800 + 2700);
    output->brightness = (uint8_t)(fusion_out[1] * 100);
    
    return 0;
}
```

## 六、架构深水区与解决方案

> **重要**：本章档记录架构设计中的三个关键技术深水区及其解决方案，直接影响系统稳定性和可靠性。

### 6.1 深水区一：音频处理的"算力黑洞"与 VAD 兜底

#### 问题描述

规划中提到通过 ESP-DSP 库提取 MFCC，然后送入 CNN。即使 ESP32-S3 有向量加速，连续不断地计算 16kHz 音频的 FFT、梅尔滤波和 DCT，然后再跑 CNN，会极大地挤占 CPU 资源。如果网关同时还要处理 Bluetooth Mesh 的心跳和路由转发，极易导致网络协议栈超时断连。

#### 解决方案：VAD（语音活动检测）唤醒机制

**架构改进**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    音频处理流水线（改进后）                                 │
│                                                                         │
│  ┌───────────────┐   ┌───────────────┐   ┌───────────────┐             │
│  │  INMP441 麦克风 │   │   VAD 检测     │   │  MFCC + CNN   │             │
│  │  16kHz I2S    │──▶│  能量/过零率   │──▶│  精细分类      │             │
│  └───────────────┘   └───────┬───────┘   └───────────────┘             │
│                              │                                          │
│                      ┌───────▼───────┐                                 │
│                      │  唤醒触发器    │                                 │
│                      │  能量突变检测  │                                 │
│                      └───────────────┘                                 │
│                                                                         │
│  日常状态：只跑 VAD（<1% CPU）                                            │
│  唤醒状态：启动 MFCC+CNN（持续 1 秒）                                      │
│  算力节省：>80%                                                           │
└─────────────────────────────────────────────────────────────────────────┘
```

**VAD 算法选择**：

| 算法 | 算力消耗 | 检测延迟 | 适用场景 |
|------|----------|----------|----------|
| **能量阈值** | 极低 (<0.5%) | <10ms | 推荐，简单可靠 |
| 过零率检测 | 低 (~1%) | <10ms | 辅助，区分周期信号 |
| WebRTC VAD | 中 (~5%) | ~30ms | 高精度场景 |

**推荐实现**：基于短时能量 + 过零率的组合检测

```c
// prj/include/audio/vad.h
typedef struct {
    float energy_threshold;      // 能量阈值（自适应）
    float zcr_threshold;         // 过零率阈值
    uint16_t hangover_frames;    // 拖尾帧数（避免断续）
    uint16_t silence_frames;     // 静音帧计数
    bool is_active;              // 当前状态
} vad_config_t;

/**
 * @brief VAD 检测主函数
 * @param audio 输入音频帧（16kHz, 10ms = 160 samples）
 * @return true 检测到语音活动，需唤醒 MFCC+CNN
 */
bool vad_detect(int16_t *audio, size_t len, vad_config_t *cfg);
```

**自适应阈值机制**：
- 系统启动时统计环境噪声基线
- 运行时动态调整阈值（避免固定阈值失效）
- 检测到持续静音后降低阈值（省电）

### 6.2 深水区二：多模态时序对齐（Time Synchronization）

#### 问题描述

雷达 LD2410 走 UART，帧率约 10Hz（每 100ms 一帧）。而声音特征（32帧 MFCC）可能代表过去 320ms 到 500ms 的声音切片。如果代码中只是简单地把"最新收到的 1 帧雷达"和"刚刚算出的 1 次音频分类"拼在一起送给融合层，它们在物理世界中可能不是同一时刻发生的（比如门响了和人进来的时间差）。

#### 解决方案：带时间戳的环形缓冲区（Ring Buffer）

**架构设计**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    时序对齐架构                                           │
│                                                                         │
│  雷达数据流 (10Hz)              音频数据流 (触发式)                         │
│  ┌───────────────┐              ┌───────────────┐                       │
│  │  LD2410 UART  │              │  INMP441 I2S  │                       │
│  │  t=0,100,200ms│              │  t=0~500ms    │                       │
│  └───────┬───────┘              └───────┬───────┘                       │
│          │                              │                                │
│          ▼                              ▼                                │
│  ┌───────────────┐              ┌───────────────┐                       │
│  │ 雷达环形缓冲区 │              │ 音频特征缓冲区 │                       │
│  │ 10帧 × 时间戳 │              │ 最近 1 秒特征 │                       │
│  │ + 能量趋势    │              │ + 时间戳      │                       │
│  └───────┬───────┘              └───────┬───────┘                       │
│          │                              │                                │
│          └──────────────┬───────────────┘                                │
│                         ▼                                                │
│                 ┌───────────────┐                                       │
│                 │  时序对齐器    │                                       │
│                 │  找最佳匹配帧  │                                       │
│                 └───────┬───────┘                                       │
│                         ▼                                                │
│                 ┌───────────────┐                                       │
│                 │  融合模型输入  │                                       │
│                 │  19维对齐特征  │                                       │
│                 └───────────────┘                                       │
└─────────────────────────────────────────────────────────────────────────┘
```

**数据结构定义**：

```c
// prj/include/common/timestamped_buffer.h

typedef struct {
    int64_t timestamp_us;    // 微秒级时间戳（来自 esp_timer）
    float data[8];           // 雷达 8 维特征
} radar_sample_t;

typedef struct {
    int64_t timestamp_us;    // 微秒级时间戳
    float features[5];       // 声音分类 5 维概率
} sound_sample_t;

// 雷达环形缓冲区：存储最近 1 秒数据（10 帧）
typedef struct {
    radar_sample_t buffer[10];
    uint8_t head;
    uint8_t count;
    pthread_mutex_t mutex;   // 线程安全
} radar_ring_buffer_t;

// 能量趋势计算（融合模型额外输入）
typedef struct {
    float energy_mean;       // 过去 1 秒能量均值
    float energy_variance;   // 能量方差（区分静止/运动）
    float energy_trend;      // 能量趋势（进入/离开）
} radar_trend_t;
```

**时序对齐算法**：

```c
/**
 * @brief 找到最接近音频时间戳的雷达帧
 * @param radar_buf 雷达环形缓冲区
 * @param sound_ts 音频特征时间戳
 * @param out 输出对齐后的雷达特征
 * @return 0 成功，-1 无匹配数据
 */
int align_radar_sound(radar_ring_buffer_t *radar_buf, 
                      int64_t sound_ts,
                      float *out_features);
```

**融合层输入扩展**：

| 原方案 | 改进方案 | 说明 |
|--------|----------|------|
| 声音 5 维 | 声音 5 维 | 不变 |
| 雷达 6 维 | 雷达 6 维 + 趋势 3 维 | 新增能量趋势 |
| 环境 8 维 | 环境 8 维 | 不变 |
| **总计 19 维** | **总计 22 维** | 扩展输入维度 |

### 6.3 深水区三：TFLM 多模型 Arena 分配

#### 问题描述

三个独立模型：声音分类器（~30KB）、雷达分析器（~2KB）、融合决策器（~3KB）。TensorFlow Lite Micro 的核心设计是静态分配一片内存（Tensor Arena）给模型运行。如果让三个模型"共享"同一个 Arena 以节省内存，它们的生命周期管理会极其复杂，极易导致内存踩踏（Memory Corruption）导致设备无限重启。

#### 解决方案：独立 Arena 分配（大力出奇迹）

**核心原则**：既然有 8MB PSRAM，绝不搞内存复用！

**内存分配策略**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    PSRAM 内存布局（共 8MB）                                │
├─────────────────────────────────────────────────────────────────────────┤
│  地址范围              │ 大小    │ 用途                                  │
├────────────────────────┼─────────┼───────────────────────────────────────┤
│  0x3C000000 - 0x3C010000 │ 64KB   │ 音频 DMA 缓冲区（16kHz × 2s）           │
│  0x3C010000 - 0x3C020000 │ 64KB   │ 雷达历史数据（10Hz × 60s × 8 维）        │
│  0x3C020000 - 0x3C040000 │ 128KB  │ 声音分类器 Tensor Arena                 │
│  0x3C040000 - 0x3C060000 │ 128KB  │ 雷达分析器 Tensor Arena                 │
│  0x3C060000 - 0x3C080000 │ 128KB  │ 融合决策器 Tensor Arena                 │
│  0x3C080000 - 0x3C800000 │ ~7.5MB │ 预留（未来扩展、端侧学习数据集）          │
└────────────────────────┴─────────┴───────────────────────────────────────┘
```

**代码实现**：

```c
// prj/src/logic/tflm_manager.c

#include "tensorflow/lite/micro/micro_interpreter.h"
#include "tensorflow/lite/micro/system_setup.h"

// 三个独立的 Tensor Arena（在 PSRAM 中分配）
static uint8_t *sound_arena = NULL;    // 声音分类器 Arena
static uint8_t *radar_arena = NULL;   // 雷达分析器 Arena  
static uint8_t *fusion_arena = NULL;  // 融合决策器 Arena

// 三个独立的 MicroInterpreter 实例
static tflite::MicroInterpreter *sound_interpreter = NULL;
static tflite::MicroInterpreter *radar_interpreter = NULL;
static tflite::MicroInterpreter *fusion_interpreter = NULL;

// Arena 大小（保守估计，实际可用 ~30KB 模型 + 中间结果）
#define SOUND_ARENA_SIZE   (128 * 1024)   // 128KB
#define RADAR_ARENA_SIZE   (128 * 1024)   // 128KB
#define FUSION_ARENA_SIZE  (128 * 1024)   // 128KB

int tflm_manager_init(void)
{
    // 在 PSRAM 中分配三个独立 Arena
    sound_arena = (uint8_t*)heap_caps_malloc(SOUND_ARENA_SIZE, MALLOC_CAP_SPIRAM);
    radar_arena = (uint8_t*)heap_caps_malloc(RADAR_ARENA_SIZE, MALLOC_CAP_SPIRAM);
    fusion_arena = (uint8_t*)heap_caps_malloc(FUSION_ARENA_SIZE, MALLOC_CAP_SPIRAM);
    
    if (!sound_arena || !radar_arena || !fusion_arena) {
        ESP_LOGE(TAG, "Failed to allocate Tensor Arenas in PSRAM");
        return -1;
    }
    
    // 初始化三个独立的 MicroInterpreter
    // ...（详见完整实现）
    
    return 0;
}
```

**多线程并行推理**：

```c

    }
}
```

| 方案 | 内存踩踏风险 | 代码复杂度 | 并行能力 |

| **独立 Arena** | **零** | **低** | **支持** |

## 七、风险评估与缓解措施

### 7.1 技术风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| TFLM 不支持某些 CNN Op | 中 | 高 | 先在 PC 端验证算子兼容性，使用 TFLM 支持列表中的 Op |
| 量化后精度大幅下降 | 低 | 中 | 使用量化感知训练（QAT），对本场景影响极小 |
| 多模型内存冲突 | 中 | 高 | 使用静态内存池，预分配所有模型 tensor arena |
| 麦克风噪声干扰 | 高 | 中 | 增加 VAD（语音活动检测），只在有声音时处理 |
| 雷达多目标干扰 | 中 | 中 | 使用距离门过滤，只关注有效区域目标 |

### 7.2 工程风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 硬件改版周期长 | 高 | 高 | 先用开发板验证，再定制 PCB |
| 数据采集困难 | 中 | 中 | 合成数据 + 小规模真实数据验证 |
| 模型训练数据不足 | 中 | 中 | 数据增强 + 迁移学习 |
| 端侧调试复杂 | 高 | 中 | 完善日志系统，支持远程诊断 |

### 7.3 产品风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 用户隐私担忧 | 中 | 高 | 明确声明本地处理，不上传音频 |
| 成本增加 | 高 | 中 | LD2410 + INMP441 约 ¥65，可接受 |
| 场景识别不准 | 中 | 高 | 设置手动切换按钮，兜底保障 |
| 响应延迟 | 低 | 中 | 优化推理流程，目标 < 200ms |
