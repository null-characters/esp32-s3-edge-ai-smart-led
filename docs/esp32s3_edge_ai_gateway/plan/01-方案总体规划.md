# ESP32-S3 会议室智能照明系统 - 方案总体规划书

## 1. 项目概述

### 1.1 项目背景

传统会议室照明系统依赖简单的定时开关或人工调节，无法根据实际环境变化（天气、时间、人员）自动优化。本项目基于ESP32-S3边缘AI网关，构建一套多维度感知的智能照明控制系统，实现：

- **环境自适应**：根据天气（晴/阴）、日落时间自动调节光强
- **人员感知**：PIR传感器检测有人/无人，自动开关灯
- **智能决策**：神经网络模型替代繁琐规则，输出平滑连续的调光策略

### 1.2 项目目标

| 目标项 | 描述 |
|--------|------|
| 核心功能 | 实现基于天气+日落时间+人员存在的智能调光 |
| 技术验证 | 验证边缘AI在嵌入式设备上的推理能力 |
| 扩展预留 | 保留BLE接口，支持未来手机App接入 |

### 1.3 预期成果

- 完整的嵌入式软件系统（Zephyr RTOS）
- 可运行的TFLite Micro神经网络模型
- 硬件驱动（UART、GPIO、WiFi）
- 端到端调光演示（自动开关灯、天气补偿、日落渐变）

### 2.2 核心子系统

| 子系统 | 功能 | 技术方案 |

| 数据采集 | 获取天气、日落时间、人员状态 | WiFi HTTP + GPIO中断 |
| 特征工程 | 7维特征向量生成 | 数学计算 + 时间编码 |

| 数据记录 | 记录照明事件 | PSRAM环形缓冲区 |

## 3. 核心功能特性

### 3.1 天气感知调光

**场景**：阴天自然光不足，需要增强人工照明

| 天气 | 编码 | 光强补偿 |
|------|------|----------|
| 晴 | 0.0 | 基础光强60% |
| 少云 | 0.25 | 基础+6% |
| 多云 | 0.5 | 基础+12% |
| 阴 | 0.75 | 基础+18% |
| 雨/雪 | 1.0 | 基础+25% |

### 3.2 日落渐变调光

**场景**：太阳下山前自然光逐渐减弱，人工照明需渐变增强

```
日落时间轴:
14:00 ────► 16:30 ────► 18:30 ────► 20:30 ────► 22:00
            │           │           │           │
            │       日落时间       日落后      深夜限幅
            │           │           │           │
sunset_proximity:
  0.0   →   0.25  →   1.0    →   1.0    →   0.5
            
光强变化:
  60%   →   65%   →   75%    →   70%    →   50%(限幅)
```

### 3.3 人员感知开关

- **有人进入**：PIR检测到高电平 → 触发开灯（AI推理确定参数）
- **持续有人**：每30秒AI推理更新（适应时间/天气变化）
- **无人离开**：PIR低电平持续5分钟 → 强制关灯

### 3.4 神经网络优势

| 对比项 | 传统规则 | 神经网络 |
|--------|----------|----------|
| 多维度输入 | 大量if-else | 天然支持7维输入 |
| 输出平滑性 | 阶梯式跳变 | 连续渐变过渡 |
| 规则维护 | 复杂难扩展 | 数据驱动训练 |
| 个性化 | 不支持 | 可学习用户偏好 |

## 4. 技术方案

### 4.1 硬件平台

| 组件 | 型号/规格 | 用途 |
|------|-----------|------|
| 主控 | ESP32-S3-N16R8 | WiFi + AI推理 + 控制 |
| 传感器 | PIR人体传感器 | 人员检测 |
| 通信 | UART2 @ 115200 | 与驱动板通信 |
| 存储 | 8MB PSRAM | 模型权重 + 数据记录 |

### 4.2 软件栈

| 层级 | 技术选型 |
|------|----------|
| 操作系统 | Zephyr RTOS |
| 网络协议 | WiFi STA + HTTP Client + SNTP |
| AI框架 | TensorFlow Lite Micro |
| 数据格式 | JSON (HTTP) / CBOR (BLE预留) |
| 构建工具 | West + CMake |

### 4.3 AI模型

| 参数 | 值 |
|------|-----|
| 网络结构 | MLP: 7→32→16→2 |
| 激活函数 | ReLU (隐藏层), Linear (输出层) |
| 参数量 | ~1.2K |
| 模型大小 | ~4KB (INT8量化) |
| 推理延迟 | <5ms @ 240MHz |

## 5. 实施阶段规划

### 5.1 总体时间线

```
Week 1-2          Week 3-4          Week 5-6
  │                 │                 │
  ▼                 ▼                 ▼

```

### 5.2 阶段目标

**Phase 1：基础硬件与通信（2周）**
- 完成硬件连接与Zephyr工程搭建
- 实现UART与驱动板通信
- 实现PIR传感器驱动
- 规则驱动的开关灯逻辑

**Phase 2：WiFi数据获取（2周）**
- WiFi连接与HTTP客户端
- 天气API集成与解析
- 日出日落时间获取
- 天气感知和日落渐变调光

**Phase 3：AI推理集成（2周）**
- PC端模型训练与量化
- TFLite Micro集成
- AI推理线程实现
- 端到端智能调光验证

## 7. 风险与对策

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| UART通信不稳定 | 中 | 控制失败 | 添加重试机制、CRC校验 |
| WiFi连接失败 | 中 | 无法获取天气 | 使用默认配置、定期重连 |
| TFLM内存不足 | 低 | 推理失败 | 简化模型、优化内存布局 |
| PIR误触发 | 中 | 频繁开关灯 | 软件消抖、持续检测 |
| INT8量化精度损失 | 低 | 预测偏差 | 混合量化、增加训练数据 |
