# v4 规划：基于 ESP-Claw 的云端混合智能体升级

> **定位**：作为 v3 (离线交互) 的上位增强包。在保持本地感知和控制的基础上，引入大模型（LLM）实现意图识别、长效记忆和自动化脚本生成。

## 一、 核心架构：云端混合 AI 驱动

在 v4 中，设备将从"命令执行器"转变为"主动智能体（Agent）"。

### 1.1 双层思维模型
* **第一反应（本地 L1）**：保留 v3 的离线语音和雷达算法。处理高频、即时、对隐私敏感的控制（如：开关灯、人走灯灭）。
* **深思熟虑（云端 L2）**：通过 **ESP-Claw** 连接大模型（如 DeepSeek / GPT-4o）。处理模糊意图、复杂逻辑编排和用户习惯学习。

---

## 二、 ESP-Claw 核心特性集成

### 2.1 五维记忆系统 (Memory System)
利用 ESP-Claw 内置的记忆框架，实现设备"越用越聪明"：
* **Profile（用户画像）**：识别当前使用者是谁（通过语音特征或手机连接状态）。
* **Preference（偏好）**：记录"用户 1 喜欢开会时色温冷一点"。
* **Fact（事实）**：记录"该会议室通常下午 2 点光照最强"。
* **Rule（自我生成的规则）**：**核心卖点**。大模型根据用户反馈，自动生成 Lua 脚本并下发到 ESP32 运行，无需重新烧录固件。

### 2.2 工具调用 (Tool Use / Function Calling)
将底层驱动能力抽象为 Agent 的工具箱：
* `set_lighting(brightness, color_temp)`：调节光照。
* `get_sensor_data()`：获取当前雷达和多模态分类结果。
* `send_notify(message)`：通过 IM（微信/Telegram）向用户推送建议。

## 三、 v4 分阶段实施路线图

### Phase 1: 基础设施搭建 (POC)
* 在 ESP-IDF 中集成 ESP-Claw 框架。
* 配置 Wi-Fi 自动联网与 HTTPS 通信。
* 接入云端大模型 API（如 DeepSeek-V3），实现基础的"闲聊"交互。

### Phase 2: "端云协作"仲裁器升级
* **事件上报**：将 v2 的场景识别结果（如"检测到在开会"）作为 `Event` 推送给 ESP-Claw。
* **策略下发**：大模型根据 `Event` 和 `Preference` 决定是否需要覆盖本地默认逻辑。
* **延迟优化**：实现"本地预响应 + 云端精细调节"的策略，消除用户等待感。

### Phase 3: 多模态社交化 (IM Integration)
* 接入 Telegram 或飞书机器人。
* 用户可以在公司聊天群里直接询问网关："会议室 3 现在有人吗？"或"帮我把灯预约到下午 3 点的暖光模式"。

## 四、 关键对比：v3 vs v4

| 维度 | v3 (离线完全态) | v4 (云端智能体) |
| :--- | :--- | :--- |
| **响应速度** | 极快 (<200ms) | 较慢 (1-3s) |
| **隐私性** | 绝对安全 (全离线) | 相对安全 (涉及云端传输) |
| **理解力** | 固定命令词 | 自然语言理解、多轮对话 |
| **灵活性** | 逻辑固定 (C 语言) | 动态脚本生成 (Lua / Agent Rules) |
| **适用场景** | 基础照明、核心控制 | 个性化管家、复杂办公场景管理 |

## 五、 产品经理视角：体验与成本的平衡术

> 核心诉求：设备不能变傻（即使断网），而且绝不能让云端 API 账单失控。

### 5.1 解决 Token 成本：引入"意图分流路由器" (Intent Router)

**策略**：不要把用户说的每一句话都扔给大模型。当唤醒词（"小白小白"）触发后，用户的语音指令先经过本地的 MultiNet（离线命令词识别）。

**分流逻辑**：

| 场景 | 处理方式 | 响应时间 | Token 成本 |
| :--- | :--- | :--- | :--- |
| 命中本地高频指令（如"打开专注模式"、"调暗一点"） | 直接本地执行 | <200ms | 0 |
| 未命中本地指令 / 复杂句式（如"我今天要通宵加班，帮我把灯光调成适合看屏幕的，明早 8 点再叫醒我"） | 启动录音，发送给云端 ESP-Claw | 1-3s | 按实际消耗 |

**产品收益**：90% 的日常操作在本地瞬间完成体验极佳，只有 10% 的长尾复杂需求上云。

### 5.2 解决网络依赖：设计"优雅降级" (Graceful Degradation) UX

**背景**：断网是 IoT 设备的常态。用户最反感的是断网时设备变成"智障"并不断报错。

**策略**：设计明确的双态反馈机制。

| 网络状态 | 唤醒灯颜色 | 运行模式 | 语音提示 |
| :--- | :--- | :--- | :--- |
| 网络畅通 | 蓝色 | Agent 模式（云端 + 本地） | 无 |
| Wi-Fi 断开 / API 超时 (>3s) | 绿色 | v3 本地模式（雷达 + 环境音 + 离线语音） | "网络未连接，已切换至本地控制"（仅一次） |

**UX 原则**：语音提示应极度克制，绝不允许出现"网络异常请重试"这种反人类设计的死循环。

## 六、 架构师视角：榨干芯片与防范死锁

> 核心任务：把庞大的 Agent 引擎和底层的实时传感器流塞进同一个 8MB 内存池里，且绝不能发生内存踩踏。

### 6.1 解决内存压力与 DMA 争抢：冷热数据分离架构

ESP32-S3 的 8MB PSRAM 是通过 SPI 总线外挂的，速度慢；而内部 SRAM（约 512KB）速度极快，但极其稀缺。

| 数据类型 | 存储位置 | 原因 | 配置方式 |
| :--- | :--- | :--- | :--- |
| ESP-SR AEC、麦克风 I2S DMA 缓冲区 | **内部 SRAM** | 防止 Cache Miss 导致爆音和丢帧 | menuconfig 强制分配 |
| TFLM Tensor Arena、ESP-Claw HTTP Buffer、cJSON 解析树 | **外部 PSRAM** | 占用大，容忍稍慢访问 | `heap_caps_malloc(size, MALLOC_CAP_SPIRAM)` |

**内存复用策略**：当 ESP-Claw 开始进行 HTTPS 请求云端大模型时，可以临时挂起 TFLM 的场景音分类任务，释放部分 CPU 带宽和 SPI 总线带宽，防止网络超时。

### 6.2 重构优先级仲裁器 (The v4 Arbiter)

在 v3 中，设计了 `语音控制 > 自动决策` 的仲裁器。在 v4 中，需要加入一个新的幽灵节点——**Cloud Agent Rule**。

**仲裁树升级**：

```
优先级（从高到低）：
┌─────────────────────────────────────────────────────────┐
│ 1. 用户本地明确指令（"关灯"）                            │
│    ↓                                                    │
│ 2. ESP-Claw 云端下发的动态 Lua 脚本规则                  │
│    （例："如果下午 3 点且雷达检测有人，色温设为 4000K"）  │
│    ↓                                                    │
│ 3. v2 版本的本地多模态 AI 默认融合算法（兜底）           │
└─────────────────────────────────────────────────────────┘
```

### 6.3 状态机的超时自愈 (Self-Healing FSM)

网络请求很容易阻塞。ESP-Claw 呼叫 API 时，如果 5 秒没有回应，FreeRTOS 任务不能死等。

**解决方案**：
* 引入 **Watchdog（看门狗）** 和 **异步事件驱动**
* 发送请求后，网关继续用本地雷达控制灯光（不阻塞主线程）
* 当云端结果（回调函数）返回时，再以事件（Event）的形式插入到执行队列中

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  发送请求    │ ──→ │  继续本地    │ ──→ │  云端回调    │
│  (非阻塞)    │     │  雷达控制    │     │  插入队列    │
└──────────────┘     └──────────────┘     └──────────────┘
        │                   │                    │
        └───────────────────┴────────────────────┘
                    Watchdog 监控（5s 超时）
```

## 七、 实施 v4 的最安全路径 (MVP)

> ⚠️ **警告**：千万不要直接在完美的 v3 代码上修改！

### 7.1 分支开发策略

```bash
# 从 v3 代码库拉出全新的功能分支
git checkout -b feature/agent-v4
```

### 7.2 剥离测试（极简链路验证）

**目标**：先不合并雷达和本地多模态代码，只跑通以下极简链路：

```
ESP-SR 唤醒 → 录音 → 发送 ESP-Claw (云端 DeepSeek) → 云端下发控制指令 → 调整 LED
```

### 7.3 内存安全阈值测量

使用 `heap_caps_get_free_size()` 打印系统剩余内存：

| 内存类型 | 安全阈值 | 检查方式 |
| :--- | :--- | :--- |
| 内部 SRAM | > 50KB | `heap_caps_get_free_size(MALLOC_CAP_INTERNAL)` |
| 外部 PSRAM | > 2MB | `heap_caps_get_free_size(MALLOC_CAP_SPIRAM)` |

**合并条件**：只有当两项指标同时满足时，才能安全地把雷达和 TFLM 的代码合并进来。

## 八、 技术难点预警（汇总）

| 难点 | 影响 | 解决方案 |
| :--- | :--- | :--- |
| 内存压力 | ESP-Claw + ESP-SR AEC 消耗 PSRAM 带宽 | 冷热数据分离、内存复用 |
| 网络稳定性 | 断网导致设备不可用 | 优雅降级到 v3 本地模式 |
| Token 成本控制 | API 账单失控 | 意图分流路由器（90% 本地执行） |
| DMA 争抢 | 音频爆音、丢帧 | 底层音频锁在内部 SRAM |
| 网络阻塞 | 任务死等、系统卡死 | Watchdog + 异步事件驱动 |
