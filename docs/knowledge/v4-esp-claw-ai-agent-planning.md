# v4 规划：基于 ESP-Claw 的云端混合智能体升级

> **定位**：作为 v3 (离线交互) 的上位增强包。在保持本地感知和控制的基础上，引入大模型（LLM）实现意图识别、长效记忆和自动化脚本生成。

## 一、 核心架构：云端混合 AI 驱动

在 v4 中，设备将从"命令执行器"转变为"主动智能体（Agent）"。

### 1.1 双层思维模型
* **第一反应（本地 L1）**：保留 v3 的离线语音和雷达算法。处理高频、即时、对隐私敏感的控制（如：开关灯、人走灯灭）。
* **深思熟虑（云端 L2）**：通过 **ESP-Claw** 连接大模型（如 DeepSeek / GPT-4o）。处理模糊意图、复杂逻辑编排和用户习惯学习。

---

## 二、 ESP-Claw 核心特性集成

### 2.1 五维记忆系统 (Memory System)
利用 ESP-Claw 内置的记忆框架，实现设备"越用越聪明"：
* **Profile（用户画像）**：识别当前使用者是谁（通过语音特征或手机连接状态）。
* **Preference（偏好）**：记录"用户 1 喜欢开会时色温冷一点"。
* **Fact（事实）**：记录"该会议室通常下午 2 点光照最强"。
* **Rule（自我生成的规则）**：**核心卖点**。大模型根据用户反馈，自动生成 Lua 脚本并下发到 ESP32 运行，无需重新烧录固件。

### 2.2 工具调用 (Tool Use / Function Calling)
将底层驱动能力抽象为 Agent 的工具箱：
* `set_lighting(brightness, color_temp)`：调节光照。
* `get_sensor_data()`：获取当前雷达和多模态分类结果。
* `send_notify(message)`：通过 IM（微信/Telegram）向用户推送建议。

## 三、 v4 分阶段实施路线图

### Phase 1: 基础设施搭建 (POC)
* 在 ESP-IDF 中集成 ESP-Claw 框架。
* 配置 Wi-Fi 自动联网与 HTTPS 通信。
* 接入云端大模型 API（如 DeepSeek-V3），实现基础的"闲聊"交互。

### Phase 2: "端云协作"仲裁器升级
* **事件上报**：将 v2 的场景识别结果（如"检测到在开会"）作为 `Event` 推送给 ESP-Claw。
* **策略下发**：大模型根据 `Event` 和 `Preference` 决定是否需要覆盖本地默认逻辑。
* **延迟优化**：实现"本地预响应 + 云端精细调节"的策略，消除用户等待感。

### Phase 3: 多模态社交化 (IM Integration)
* 接入 Telegram 或飞书机器人。
* 用户可以在公司聊天群里直接询问网关："会议室 3 现在有人吗？"或"帮我把灯预约到下午 3 点的暖光模式"。

## 四、 关键对比：v3 vs v4

| 维度 | v3 (离线完全态) | v4 (云端智能体) |
| :--- | :--- | :--- |
| **响应速度** | 极快 (<200ms) | 较慢 (1-3s) |
| **隐私性** | 绝对安全 (全离线) | 相对安全 (涉及云端传输) |
| **理解力** | 固定命令词 | 自然语言理解、多轮对话 |
| **灵活性** | 逻辑固定 (C 语言) | 动态脚本生成 (Lua / Agent Rules) |
| **适用场景** | 基础照明、核心控制 | 个性化管家、复杂办公场景管理 |

## 五、 技术难点预警

1. **内存压力**：ESP-Claw 加上 ESP-SR 的 AEC 算法，会极大地消耗 PSRAM 带宽。需要优化内存池分配。
2. **网络稳定性**：当 Wi-Fi 断开时，系统必须能无缝降级回 v3 的纯离线模式，不能因为断网导致灯开不了。
3. **Token 成本控制**：频繁的心跳和事件上报会消耗大模型 API 额度，需要设计合理的"事件触发上报"机制，而非实时上报。
