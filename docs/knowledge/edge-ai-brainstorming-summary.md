## 议题一：云端"理想国" vs 边缘"现实派"

### 背景

在 PC/云端训练时，开发者倾向于堆叠更深的网络（如几百层的神经网络）来追求 99.9% 的准确率。但 ESP32-S3 的 SRAM 只有几百 KB，算力有限。边缘 AI 架构师必须在以下三者之间走钢丝：

- **模型精度 (Accuracy)**：模型预测的正确程度
- **推理延迟 (Latency)**：从输入到输出的耗时
- **内存占用 (Memory Footprint)**：模型运行所需的 RAM/Flash 空间

### 讨论

**Q：如何为 ESP32-S3 挑选合适的神经网络结构？为什么选 MobileNet 或简单 CNN，而不是复杂的 Transformer？**

**A：对本项目而言，以上选项全部不需要。**

本项目的输入数据是**一维数值型数据**（PIR 高低电平、时间小时数、天气阴晴程度等），不涉及图像/视频场景。处理此类数据，最合适的结构是**全连接神经网络（MLP，多层感知机）**——它是 AI 领域最基础、最成熟的结构。

```
┌─────────────────────────────────────────────────────────────────┐
│                    神经网络结构选型对照                            │
├──────────────┬──────────────────┬───────────────────────────────┤
│ 网络类型      │ 适用场景          │ 本项目适用性                    │
├──────────────┼──────────────────┼───────────────────────────────┤
│ CNN          │ 图像识别、视觉     │ ✗ 不涉及                       │
│ Transformer  │ 自然语言、序列     │ ✗ 不涉及                       │
│ RNN/LSTM     │ 时间序列预测      │ △ 可选但过重                    │
│ MLP          │ 一维数值、分类/回归 │ ✓ 最佳选择                     │
└──────────────┴──────────────────┴───────────────────────────────┘
```

### MVP 预期指标

| 指标 | 预估值 |
|------|--------|
| 模型文件大小 (.tflite) | < 5 KB |
| 单次推理耗时 | 几微秒 ~ 几毫秒 |
| 内存占用 | 极低，ESP-NN 加速甚至无需全力运转 |

### 边缘计算的核心优势

本项目作为 MVP，上云会增加开发复杂度。选择边缘计算的核心理由：

1. **隐私与安全**：传感器数据无需上传，直接在本地处理，贴合隐私敏感人群需求
2. **低延迟**：无需网络往返，推理延迟远低于云端方案
3. **离线可用**：无网环境下仍能正常工作，适合会议室、地下室等场景
4. **市场更宽**：除特别需要实时联网的场景外，边缘计算适用场景更大

## 议题二："喂"什么数据，长什么脑子

### 背景

AI 圈名言 "Garbage in, garbage out"（垃圾进，垃圾出）——训练数据决定了模型的智商上限。如果用 PC 端高清照片训练，再放到低分辨率红外传感器上推理，模型大概率失效。

### 讨论

**Q：如何保证训练数据与边缘推理时的传感器真实输入保持一致？**

**A：本项目的数据源非常清晰，不存在此问题。**

本项目的数据维度：

| 数据源 | 类型 | 取值范围 | 用途 |
|--------|------|----------|------|
| PIR 传感器 | 离散 | 0 / 1 | 人体存在检测 |
| 时间 | 连续 | 0 ~ 23（小时） | 白天/夜晚判断 |
| 天气信息 | 连续/离散 | 阴晴程度、日升日落时间 | 光强补偿 |

这些数据本身就是一维数值，PC 端训练与 ESP32 推理的数据格式完全一致，不存在"训练-推理数据分布不一致"的问题。

### 核心概念：AI 如何从数据中"学习"规则

传统编程 vs AI 编程的根本差异：

```
传统编程：规则 + 数据 ──────► 答案
    if (pir == 1 && time > 18) { brightness = 80; }

AI 编程：  答案 + 数据 ──────► 规则
    给 AI 几千条 [PIR, 时间, 天气] → [亮度] 的历史记录，
    AI 自己总结出映射规则
```

#### 1. 数据的长相

训练数据可以理解为一个巨大的 Excel 表格：

```
┌────────┬────────┬──────────┬──────────────┐
│ PIR状态 │ 小时数  │ 云量(%)   │ 目标亮度(%)   │
├────────┼────────┼──────────┼──────────────┤
│   1    │   20   │    10    │     80       │  ← 特征 (X) + 标签 (Y)
│   0    │   14   │    80    │     30       │
│   1    │    7   │    50    │     60       │
│  ...   │  ...   │   ...   │     ...      │
└────────┴────────┴──────────┴──────────────┘
   ←─── Features (X) ───→   ← Label (Y) →
```

- **特征 (Features, X)**：输入给 AI 的条件，如 `[PIR状态, 当前小时数, 云量百分比]`
- **标签 (Labels, Y)**：你希望 AI 做出的决定，如 `[目标灯光亮度]`

#### 2. 训练数据如何产生？

**方法一：人工构造（合成数据）**

编写 Python 脚本，用代码生成几千行训练数据。例如规定"晚上 8 点且 PIR 有人 → 亮度 80%"，在表格中填入对应行，并加入随机扰动增加泛化性。

```python
# 伪代码示意
for hour in range(24):
    for pir in [0, 1]:
        for cloud in range(0, 101, 10):
            brightness = rule_based_compute(pir, hour, cloud)
            brightness += random_noise()  # 加入扰动
            dataset.append([pir, hour, cloud, brightness])
```

**方法二：真实采集（行为克隆）**

将 ESP32-S3 部署在会议室运行一周，但不做智能控制。每次用户通过物理开关调节亮度时，ESP32 记录下那一刻的 `[PIR, 时间, 天气]` 以及用户设定的 `[亮度]`。收集几千条后训练，AI 就能学会该会议室用户的偏好。

```
┌───────────────────────────────────────────────────────────────┐
│                   数据产生方式对比                              │
├────────────────┬──────────────────┬───────────────────────────┤
│ 方式            │ 优点              │ 缺点                      │
├────────────────┼──────────────────┼───────────────────────────┤
│ 合成数据        │ 快速、可控、零成本  │ 规则由人定义，可能不够自然    │
│ 行为克隆        │ 真实偏好、自然决策  │ 需要采集周期、成本较高        │
│ 推荐：MVP 阶段  │ 先用合成数据快速验证 │ 后期用行为克隆优化体验        │
└────────────────┴──────────────────┴───────────────────────────┘
```

#### 3. 交互过程

训练结束后，模型变成一个"黑盒函数"：

```
亮度 = Model(PIR, 时间, 天气)
```

在 ESP32 上，只需将实时的三个传感器值传给 TFLite 引擎，它直接输出亮度值。

## 议题三：一劳永逸是不存在的

### 背景

ESP32-S3 烧录固件后，模型被冻结（静态推理）。但现实世界是动态的——夏天训练的模型到冬天可能因穿着变化导致识别率暴跌，这就是**概念漂移 (Data Drift)**。

### 讨论

**Q：既然 ESP32 无法本地重新训练，如何让它具备"持续进化"的能力？**

**A：MVP 阶段不需要复杂机制，用"见多识广"打败概念漂移。**

#### 泛化能力 (Generalization) —— 项目的核心策略

将漂移因素纳入训练数据集，最大化覆盖大多数场景。目标：覆盖 80%+ 场景即可，不必追求完美。

具体做法：在构建训练数据（上述"Excel 表格"）时，把各种极端情况都加进去：

| 场景 | PIR | 时间 | 天气 | 期望亮度 |
|------|-----|------|------|----------|
| 正常晴天白天 | 1 | 10 | 晴 | 20% |
| 正常夜晚 | 1 | 21 | 晴 | 80% |
| **暴雨如注的中午** | 1 | 12 | **暴雨** | **70%** |
| **阴天黄昏** | 0 | 17 | **阴** | **40%** |
| 深夜无人 | 0 | 3 | 晴 | 5% |

只要训练数据覆盖了这些边缘场景，模型就能泛化应对。

#### 进阶机制（列入 Roadmap，MVP 不做）

| 机制 | 说明 | 适用阶段 |
|------|------|----------|
| **OTA 模型下发** | 通过 Wi-Fi 将新训练的 .tflite 模型文件下发给 ESP32-S3，替换旧模型 | 产品化阶段 |
| **影子模式 (Shadow Mode)** | 模型在后台偷偷运行，给出预测但不实际控制，仅对比预测与人的实际操作，用于评估模型准确性 | 数据采集/评估阶段 |

```
┌───────────────────────────────────────────────────────────────┐
│                    模型演进路线图                               │
│                                                               │
│   MVP 阶段                产品化阶段          持续优化阶段       │
│   ┌──────────┐           ┌──────────┐       ┌──────────┐     │
│   │ 合成数据  │           │ 行为克隆  │       │ 影子模式  │     │
│   │ 训练     │ ────────► │ 数据采集  │ ────► │ 评估     │     │
│   │ 烧录部署  │           │ OTA 更新  │       │ 持续迭代  │     │
│   └──────────┘           └──────────┘       └──────────┘     │
│                                                               │
│   目标：覆盖 80% 场景     目标：真实偏好       目标：持续进化     │
└───────────────────────────────────────────────────────────────┘
```

## 议题四：带着镣铐跳舞

### 背景

并非所有在 TensorFlow/PyTorch 里能跑的代码都能在 ESP32-S3 上跑。TensorFlow Lite Micro (TFLM) 只支持部分核心操作符 (Ops)，如果训练时用了 TFLM 或 ESP-NN 不支持的算子，转换部署时会直接报错。

### 讨论

**Q：如何在写训练代码之前，就摸透边缘芯片的硬件底牌？**

**A：以实测为准，TDD 驱动验证。**

#### 硬件底牌：ESP32-S3 AI 加速能力

ESP32-S3 之所以被选中，正是因为其内置 AI 加速库（ESP-NN）。关键资源：

| 资源 | 规格 |
|------|------|
| SRAM | 512 KB |
| PSRAM | 可外扩 2~8 MB |
| AI 加速 | ESP-NN 库（向量运算、卷积等硬件加速） |
| 推理引擎 | TFLite Micro (TFLM) |

#### TDD 驱动的验证流程

本项目已配置 Zephyr Twister 测试框架和 `native_sim`，推荐开发流：

```
┌───────────────────────────────────────────────────────────────┐
│                    TDD 验证流程                                │
│                                                               │
│  ① Python 训练                                               │
│     └─► 生成极小 MLP 模型                                     │
│                                                               │
│  ② 转换格式                                                   │
│     └─► model.tflite (TFLite FlatBuffer)                     │
│                                                               │
│  ③ PC 端单元测试 (Twister + native_sim)                       │
│     └─► 验证 TFLM C++ API 调用正确性                           │
│     └─► 确认算子兼容性（不支持的 Op 在此阶段就会暴露）            │
│                                                               │
│  ④ 真机烧录验证 (ESP32-S3)                                    │
│     └─► 喂入伪造传感器数值，串口打印输出                          │
│     └─► 测量：推理耗时 (ms)、内存占用 (KB)、输出合理性            │
│                                                               │
│  ⑤ 回归迭代                                                   │
│     └─► 不通过则回到 ① 调整模型/数据                            │
└───────────────────────────────────────────────────────────────┘
```

**核心原则**：模拟器永远无法替代真机的内存碎片和总线时序，实测才是最终裁判。

## 结论与行动项

### 四个议题的收敛结论

| 议题 | 核心结论 | MVP 行动 |
|------|----------|----------|
| 模型选型 | 使用 MLP（全连接网络），不涉及 CNN/Transformer | 设计极简 MLP，目标 < 5KB |
| 数据策略 | 训练数据与推理数据天然一致，用合成数据快速启动 | 编写 Python 脚本生成合成数据集 |
| 概念漂移 | 通过提升泛化能力覆盖 80%+ 场景，MVP 不做 OTA/影子模式 | 在合成数据中加入极端场景 |
| 软硬协同 | TDD 驱动，从 PC 单元测试到真机实测逐步验证 | Twister + native_sim → ESP32-S3 烧录验证 |

### MVP 开发优先级

1. **编写数据生成脚本**：用 Python 生成合成训练数据（含极端场景）
2. **训练 MLP 模型**：极简结构，导出 `.tflite`
3. **PC 端 TDD 验证**：Twister 单元测试确保 TFLM API 调用正确
4. **真机实测**：烧录到 ESP32-S3，验证推理耗时和内存占用
5. **迭代优化**：根据实测结果调整模型结构和数据
