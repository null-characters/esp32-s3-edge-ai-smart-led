## 议题一：云端"理想国" vs 边缘"现实派"

### 背景

在 PC/云端训练时，开发者倾向于堆叠更深的网络（如几百层的神经网络）来追求 99.9% 的准确率。但 ESP32-S3 的 SRAM 只有几百 KB，算力有限。边缘 AI 架构师必须在以下三者之间走钢丝：

- **模型精度 (Accuracy)**：模型预测的正确程度
- **推理延迟 (Latency)**：从输入到输出的耗时
- **内存占用 (Memory Footprint)**：模型运行所需的 RAM/Flash 空间

### 讨论

**Q：如何为 ESP32-S3 挑选合适的神经网络结构？为什么选 MobileNet 或简单 CNN，而不是复杂的 Transformer？**

**A：对本项目而言，以上选项全部不需要。**

本项目的输入数据是**一维数值型数据**（PIR 高低电平、时间小时数、天气阴晴程度等），不涉及图像/视频场景。处理此类数据，最合适的结构是**全连接神经网络（MLP，多层感知机）**——它是 AI 领域最基础、最成熟的结构。

```
┌─────────────────────────────────────────────────────────────────┐
│                    神经网络结构选型对照                            │
├──────────────┬──────────────────┬───────────────────────────────┤
│ 网络类型      │ 适用场景          │ 本项目适用性                    │
├──────────────┼──────────────────┼───────────────────────────────┤
│ CNN          │ 图像识别、视觉     │ ✗ 不涉及                       │
│ Transformer  │ 自然语言、序列     │ ✗ 不涉及                       │
│ RNN/LSTM     │ 时间序列预测      │ △ 可选但过重                    │
│ MLP          │ 一维数值、分类/回归 │ ✓ 最佳选择                     │
└──────────────┴──────────────────┴───────────────────────────────┘
```

### MVP 预期指标

| 指标 | 预估值 |
|------|--------|
| 模型文件大小 (.tflite) | < 5 KB |
| 单次推理耗时 | 几微秒 ~ 几毫秒 |
| 内存占用 | 极低，ESP-NN 加速甚至无需全力运转 |

### 边缘计算的核心优势

本项目作为 MVP，上云会增加开发复杂度。选择边缘计算的核心理由：

1. **隐私与安全**：传感器数据无需上传，直接在本地处理，贴合隐私敏感人群需求
2. **低延迟**：无需网络往返，推理延迟远低于云端方案
3. **离线可用**：无网环境下仍能正常工作，适合会议室、地下室等场景
4. **市场更宽**：除特别需要实时联网的场景外，边缘计算适用场景更大

### 深入追问：MLP / CNN / Transformer 有什么区别？

这三者都是"神经网络"，但结构设计用来解决不同维度的物理问题。可以想象成三种不同特长的员工。

#### 1. MLP（全连接神经网络）—— "数据分析员"

- **特点**：最基础的结构。网络中每个神经元都和上一层的所有神经元相连（"全连接"）
- **擅长**：处理一维的、结构化的表格数据（如本项目的 PIR 状态、时间、天气）
- **缺点**：如果输入数据太大（如 1080p 图片有几百万像素），全连接会导致计算量爆炸

```
输入层          隐藏层           输出层
  ○              ○
  ○ ──────────►  ○  ──────────►  ○
  ○              ○
  ○              ○

每条线代表一个权重参数
"全连接" = 每个神经元与上一层所有神经元相连
```

#### 2. CNN（卷积神经网络）—— "视觉质检员"

- **特点**：不再"全连接"，而是用卷积核在数据上滑动扫描，提取局部特征（边缘、角点、纹理）
- **擅长**：处理二维或三维的网格数据，尤其是图像、视频（人脸识别、缺陷检测）
- **缺点**：不太擅长理解跨越很长时间段的前后上下文逻辑

```
输入图像          卷积核滑动          特征图
┌─────────┐      ┌───┐              ┌───────┐
│ ■ □ □ ■ │      │1 0│              │ ◉ ◉   │
│ □ ■ ■ □ │  ──► │0 1│ 滑动扫描  ──► │   ◉ ◉ │
│ □ ■ ■ □ │      └───┘              │       │
│ ■ □ □ ■ │                         └───────┘
└─────────┘

卷积核像"放大镜"，提取局部特征
```

#### 3. Transformer（注意力机制网络）—— "速读理解大师"

- **特点**：核心是"自注意力（Self-Attention）"机制，看当前数据时会计算它与上下文中所有其他数据的关联权重。ChatGPT 即基于此架构
- **擅长**：处理带强前后顺序和上下文的序列数据（自然语言处理、语音识别、长时间序列预测）
- **缺点**：极其吃算力和内存，通常边缘 MCU（如 ESP32）极难运行完整 Transformer

```
"我 爱 北京 天安门"
  ↓   ↓    ↓     ↓
  ●───●────●─────●
  │\ /│\  /│\  /│
  │ ● │ ● │ ● │ ●  ← 每个词都与其他所有词计算关联权重
  │/ \|/  \|/  \|/
  ●───●────●─────●

"注意力" = 每个位置都关注全局上下文
```

#### 对比总结

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    三大神经网络结构对比                                    │
├──────────────┬────────────────────┬────────────────┬─────────────────────┤
│ 结构          │ 核心机制            │ 擅长场景        │ 边缘 MCU 可行性      │
├──────────────┼────────────────────┼────────────────┼─────────────────────┤
│ MLP          │ 全连接（Dense）     │ 一维表格数据    │ ✓ 完美适配           │
│ CNN          │ 卷积（Convolution） │ 图像/视频      │ △ 需精简模型         │
│ Transformer  │ 自注意力（Attention）│ 语言/长序列    │ ✗ 算力内存不足        │
└──────────────┴────────────────────┴────────────────┴─────────────────────┘
```

**结论**：对于本智能照明 MVP，**MLP 是唯一且最完美的解**。

## 议题二："喂"什么数据，长什么脑子

### 背景

AI 圈名言 "Garbage in, garbage out"（垃圾进，垃圾出）——训练数据决定了模型的智商上限。如果用 PC 端高清照片训练，再放到低分辨率红外传感器上推理，模型大概率失效。

### 讨论

**Q：如何保证训练数据与边缘推理时的传感器真实输入保持一致？**

**A：本项目的数据源非常清晰，不存在此问题。**

本项目的数据维度：

| 数据源 | 类型 | 取值范围 | 用途 |
|--------|------|----------|------|
| PIR 传感器 | 离散 | 0 / 1 | 人体存在检测 |
| 时间 | 连续 | 0 ~ 23（小时） | 白天/夜晚判断 |
| 天气信息 | 连续/离散 | 阴晴程度、日升日落时间 | 光强补偿 |

这些数据本身就是一维数值，PC 端训练与 ESP32 推理的数据格式完全一致，不存在"训练-推理数据分布不一致"的问题。

### 核心概念：AI 如何从数据中"学习"规则

传统编程 vs AI 编程的根本差异：

```
传统编程：规则 + 数据 ──────► 答案
    if (pir == 1 && time > 18) { brightness = 80; }

AI 编程：  答案 + 数据 ──────► 规则
    给 AI 几千条 [PIR, 时间, 天气] → [亮度] 的历史记录，
    AI 自己总结出映射规则
```

#### 1. 数据的长相

训练数据可以理解为一个巨大的 Excel 表格：

```
┌────────┬────────┬──────────┬──────────────┐
│ PIR状态 │ 小时数  │ 云量(%)   │ 目标亮度(%)   │
├────────┼────────┼──────────┼──────────────┤
│   1    │   20   │    10    │     80       │  ← 特征 (X) + 标签 (Y)
│   0    │   14   │    80    │     30       │
│   1    │    7   │    50    │     60       │
│  ...   │  ...   │   ...   │     ...      │
└────────┴────────┴──────────┴──────────────┘
   ←─── Features (X) ───→   ← Label (Y) →
```

- **特征 (Features, X)**：输入给 AI 的条件，如 `[PIR状态, 当前小时数, 云量百分比]`
- **标签 (Labels, Y)**：你希望 AI 做出的决定，如 `[目标灯光亮度]`

#### 2. 训练数据如何产生？

**方法一：人工构造（合成数据）**

编写 Python 脚本，用代码生成几千行训练数据。例如规定"晚上 8 点且 PIR 有人 → 亮度 80%"，在表格中填入对应行，并加入随机扰动增加泛化性。

```python
# 伪代码示意
for hour in range(24):
    for pir in [0, 1]:
        for cloud in range(0, 101, 10):
            brightness = rule_based_compute(pir, hour, cloud)
            brightness += random_noise()  # 加入扰动
            dataset.append([pir, hour, cloud, brightness])
```

**方法二：真实采集（行为克隆）**

将 ESP32-S3 部署在会议室运行一周，但不做智能控制。每次用户通过物理开关调节亮度时，ESP32 记录下那一刻的 `[PIR, 时间, 天气]` 以及用户设定的 `[亮度]`。收集几千条后训练，AI 就能学会该会议室用户的偏好。

```
┌───────────────────────────────────────────────────────────────┐
│                   数据产生方式对比                              │
├────────────────┬──────────────────┬───────────────────────────┤
│ 方式            │ 优点              │ 缺点                      │
├────────────────┼──────────────────┼───────────────────────────┤
│ 合成数据        │ 快速、可控、零成本  │ 规则由人定义，可能不够自然    │
│ 行为克隆        │ 真实偏好、自然决策  │ 需要采集周期、成本较高        │
│ 推荐：MVP 阶段  │ 先用合成数据快速验证 │ 后期用行为克隆优化体验        │
└────────────────┴──────────────────┴───────────────────────────┘
```

#### 3. 交互过程

训练结束后，模型变成一个"黑盒函数"：

```
亮度 = Model(PIR, 时间, 天气)
```

在 ESP32 上，只需将实时的三个传感器值传给 TFLite 引擎，它直接输出亮度值。

### 深入追问：二维表 vs C 语言数组 vs If-Else vs AI 推理

**Q：如果用 C 语言生成一个巨大的多维数组，是否也能实现同样效果？这和 AI 推理有什么区别？和一堆 if-else 堆砌的函数有什么区别？**

这个问题触及了传统编程和 AI 编程的分水岭。

#### 方案一：查表法（C 语言多维数组）

如果用 C 语言的大数组存储所有情况：

```c
// 穷举所有组合的查表法
uint8_t brightness_table[24][2][11] = {
    // hour=0, pir=0/1, cloud=0~100(步长10)
    {{5, 5, 5, ...}, {80, 80, 75, ...}},
    // hour=1, ...
};
```

**问题一：无法内插**

假设表里只有"时间=18, PIR=1 → 亮度 80"和"时间=19, PIR=1 → 亮度 90"。如果当前时间是 18.5（18 点半），查表法会卡壳，需要手动写复杂的插值算法。

**问题二：维度灾难**

只有 3 个特征时，穷举法可能只需几百行的数组。但如果接入 10 个传感器（温度、湿度、光强、人数、噪音等），需要穷举所有组合，数组会呈指数级爆炸：

```
特征数 N，每个特征取值 M 种
数组大小 = M^N

3 个特征 × 100 种取值 = 1,000,000 条
10 个特征 × 10 种取值 = 10,000,000,000 条（100 亿条！）
```

ESP32 的 Flash 根本装不下。

#### 方案二：If-Else 堆砌

```c
if (pir == 1 && time > 18 && time < 22) {
    if (cloud > 50) brightness = 70;
    else brightness = 80;
} else if (pir == 0) {
    brightness = 5;
} else if (...) {
    // 无穷无尽的分支...
}
```

**问题**：规则写死，无法适应新场景；分支复杂后维护噩梦；无法处理边界模糊情况。

#### 方案三：AI 推理（函数拟合）

**AI 推理的本质是"函数拟合"**——AI 并不是把那个巨大的"二维训练表"塞进芯片里！

在训练过程中，AI 通过数学运算（梯度下降），将几千甚至几百万行的表格**压缩成一个带权重参数的方程式**：

```
y = W1*x1 + W2*x2 + W3*x3 + b

亮度 = 0.3*PIR + 2.5*时间 - 0.1*云量 + 10
（示意，实际是非线性激活函数组合）
```

#### 三种方案对比

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    三种方案的本质区别                                      │
├──────────────┬────────────────────┬────────────────┬─────────────────────┤
│ 维度          │ 查表法              │ If-Else        │ AI 推理              │
├──────────────┼────────────────────┼────────────────┼─────────────────────┤
│ 存储方式      │ 存原始数据          │ 存规则逻辑      │ 存权重参数（公式系数）  │
│ 空间复杂度    │ O(M^N) 指数爆炸     │ O(N) 线性      │ O(N) 固定大小        │
│ 内插能力      │ 无，需手动编码       │ 无，掉入 default│ 有，自动平滑预测      │
│ 新场景适应    │ 需重新穷举          │ 需加新分支      │ 重新训练即可          │
│ 边界模糊处理  │ 无法处理            │ 难以处理        │ 天然支持             │
└──────────────┴────────────────────┴────────────────┴─────────────────────┘
```

**核心区别总结**：

1. **极度节省空间**：无论用 1 万行还是 100 万行 Excel 数据训练，最终提取的模型大小是固定的（比如 5KB 的数学公式系数）
2. **具备推断能力**：即便输入训练表里从未有过的数值（如时间 18.3），模型也能通过学到的函数曲线平滑计算出合理值（如亮度 83），无需写任何额外 `if-else`

## 议题三：一劳永逸是不存在的

### 背景

ESP32-S3 烧录固件后，模型被冻结（静态推理）。但现实世界是动态的——夏天训练的模型到冬天可能因穿着变化导致识别率暴跌，这就是**概念漂移 (Data Drift)**。

### 讨论

**Q：既然 ESP32 无法本地重新训练，如何让它具备"持续进化"的能力？**

**A：MVP 阶段不需要复杂机制，用"见多识广"打败概念漂移。**

#### 泛化能力 (Generalization) —— 项目的核心策略

将漂移因素纳入训练数据集，最大化覆盖大多数场景。目标：覆盖 80%+ 场景即可，不必追求完美。

具体做法：在构建训练数据（上述"Excel 表格"）时，把各种极端情况都加进去：

| 场景 | PIR | 时间 | 天气 | 期望亮度 |
|------|-----|------|------|----------|
| 正常晴天白天 | 1 | 10 | 晴 | 20% |
| 正常夜晚 | 1 | 21 | 晴 | 80% |
| **暴雨如注的中午** | 1 | 12 | **暴雨** | **70%** |
| **阴天黄昏** | 0 | 17 | **阴** | **40%** |
| 深夜无人 | 0 | 3 | 晴 | 5% |

只要训练数据覆盖了这些边缘场景，模型就能泛化应对。

#### 进阶机制（列入 Roadmap，MVP 不做）

| 机制 | 说明 | 适用阶段 |
|------|------|----------|
| **OTA 模型下发** | 通过 Wi-Fi 将新训练的 .tflite 模型文件下发给 ESP32-S3，替换旧模型 | 产品化阶段 |
| **影子模式 (Shadow Mode)** | 模型在后台偷偷运行，给出预测但不实际控制，仅对比预测与人的实际操作，用于评估模型准确性 | 数据采集/评估阶段 |

```
┌───────────────────────────────────────────────────────────────┐
│                    模型演进路线图                               │
│                                                               │
│   MVP 阶段                产品化阶段          持续优化阶段       │
│   ┌──────────┐           ┌──────────┐       ┌──────────┐     │
│   │ 合成数据  │           │ 行为克隆  │       │ 影子模式  │     │
│   │ 训练     │ ────────► │ 数据采集  │ ────► │ 评估     │     │
│   │ 烧录部署  │           │ OTA 更新  │       │ 持续迭代  │     │
│   └──────────┘           └──────────┘       └──────────┘     │
│                                                               │
│   目标：覆盖 80% 场景     目标：真实偏好       目标：持续进化     │
└───────────────────────────────────────────────────────────────┘
```

### 深入追问：泛化能力是"以量取胜"吗？

**Q：是否不断添加泛化条件，模型就越接近 100% 准确度？这实际是"以量取胜"？**

**A：不是以量取胜，而是以"多样性（多样的数据分布）"取胜。**

#### 无效的努力

如果收集了 10 万条"晴天且有人"的数据，模型对这种情况滚瓜烂熟，但这无助于提升它在"雨天且无人"时的判断。

```
训练数据分布示意（错误示范）：

晴天+有人：████████████████████████ 100,000 条（重复且无效）
雨天+无人：██ 200 条
阴天+有人：███ 500 条

模型学会的：只在晴天+有人时准确，其他场景瞎猜
```

#### 过拟合（Overfitting）风险

AI 圈有一个致命问题：如果过度追求训练集上的 100% 准确度，模型可能会变成"书呆子"——它死记硬背训练数据里的随机噪声（比如某天传感器电压不稳导致的异常值），导致在实际应用中遇到新情况时反而错误百出。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    过拟合 vs 欠拟合 vs 刚好                               │
│                                                                         │
│   训练集准确度   测试集准确度   状态                                       │
│       99%           60%       过拟合（死记硬背）                           │
│       70%           68%       欠拟合（学得太少）                           │
│       92%           90%       刚好（学到规律）                            │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 泛化的真谛

架构师要的是模型学到"底层规律"（如：环境光越暗 + 有人 = 灯越亮），而不是死记硬背历史数据。

**正确做法**：涵盖主要的边界条件（Edge Cases），模型就能自己填补中间的空白。

```
训练数据分布示意（正确示范）：

晴天+白天+有人：███ 1,000 条
晴天+夜晚+有人：███ 1,000 条
雨天+白天+有人：███ 1,000 条
雨天+夜晚+有人：███ 1,000 条
阴天+黄昏+无人：███ 1,000 条
暴雨+中午+有人：███ 1,000 条（极端场景）
...

每种场景均衡覆盖，模型学会底层规律
```

**结论**：80% 的高质量场景数据（覆盖主要边界条件），足以通过 AI 拟合出 99% 的场景体验。**质量 > 数量，多样性 > 重复性**。

## 议题四：带着镣铐跳舞

### 背景

并非所有在 TensorFlow/PyTorch 里能跑的代码都能在 ESP32-S3 上跑。TensorFlow Lite Micro (TFLM) 只支持部分核心操作符 (Ops)，如果训练时用了 TFLM 或 ESP-NN 不支持的算子，转换部署时会直接报错。

### 讨论

**Q：如何在写训练代码之前，就摸透边缘芯片的硬件底牌？**

**A：以实测为准，TDD 驱动验证。**

#### 硬件底牌：ESP32-S3 AI 加速能力

ESP32-S3 之所以被选中，正是因为其内置 AI 加速库（ESP-NN）。关键资源：

| 资源 | 规格 |
|------|------|
| SRAM | 512 KB |
| PSRAM | 可外扩 2~8 MB |
| AI 加速 | ESP-NN 库（向量运算、卷积等硬件加速） |
| 推理引擎 | TFLite Micro (TFLM) |

#### TDD 驱动的验证流程

本项目已配置 Zephyr Twister 测试框架和 `native_sim`，推荐开发流：

```
┌───────────────────────────────────────────────────────────────┐
│                    TDD 验证流程                                │
│                                                               │
│  ① Python 训练                                               │
│     └─► 生成极小 MLP 模型                                     │
│                                                               │
│  ② 转换格式                                                   │
│     └─► model.tflite (TFLite FlatBuffer)                     │
│                                                               │
│  ③ PC 端单元测试 (Twister + native_sim)                       │
│     └─► 验证 TFLM C++ API 调用正确性                           │
│     └─► 确认算子兼容性（不支持的 Op 在此阶段就会暴露）            │
│                                                               │
│  ④ 真机烧录验证 (ESP32-S3)                                    │
│     └─► 喂入伪造传感器数值，串口打印输出                          │
│     └─► 测量：推理耗时 (ms)、内存占用 (KB)、输出合理性            │
│                                                               │
│  ⑤ 回归迭代                                                   │
│     └─► 不通过则回到 ① 调整模型/数据                            │
└───────────────────────────────────────────────────────────────┘
```

**核心原则**：模拟器永远无法替代真机的内存碎片和总线时序，实测才是最终裁判。

### 深入追问：量化的风险与体积变化

**Q：推理完成的模型还需要量化才能导入芯片，量化有什么风险？训练模型 .tflite 只有 5KB，量化后大小还是 5KB 吗？**

**A：量化一定会改变体积，且通常是极大幅度的缩减。**

#### 什么是量化？

在云端训练时，模型的计算公式使用 32 位浮点数（Float32，占 4 字节），如权重 `0.12345678`。为了让微控制器跑得快且省内存，导出 `.tflite` 前会将浮点数映射为 8 位整数（INT8，占 1 字节）。

```
量化过程示意：

Float32 (4 字节)                    INT8 (1 字节)
┌────────────────┐                  ┌──────────┐
│ 0.12345678     │  ── 映射 ──►    │   12     │
│ 0.56789012     │  ── 映射 ──►    │   57     │
│ -0.34567890    │  ── 映射 ──►    │  -35     │
│ 0.98765432     │  ── 映射 ──►    │   99     │
└────────────────┘                  └──────────┘
  精度：小数点后 8 位               精度：整数 -128 ~ 127
  体积：4×                         体积：1×
```

#### 体积变化

因为数据类型从 4 字节变为 1 字节，**模型体积直接缩小为原来的 1/4**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    量化前后体积对比                                       │
├──────────────────────┬────────────────────┬──────────────────────────────┤
│ 阶段                  │ 数据类型            │ 模型大小                     │
├──────────────────────┼────────────────────┼──────────────────────────────┤
│ 云端训练 (H5/SavedModel) │ Float32 (4字节)   │ ~20 KB                      │
│ 转换后 (未量化 .tflite)   │ Float32 (4字节)   │ ~20 KB                      │
│ 量化后 (INT8 .tflite)    │ INT8 (1字节)      │ ~5 KB                       │
└──────────────────────┴────────────────────┴──────────────────────────────┘
```

所以，之前说的"5KB"其实是指**量化后**的 .tflite 文件大小，量化前的 Float32 版本约为 20KB。

#### 量化的风险：精度掉点

将 `0.12345678` 强行映射为 `-128 ~ 127` 之间的整数，必然会丢失精度。对于非常复杂、要求极高的模型（如医疗影像分析），量化可能导致准确率明显下降。

```
量化精度损失示意：

原始预测：亮度 = 80.52%
量化预测：亮度 = 81%
误差：0.48%  ← 对本项目而言完全可忽略
```

#### 对本项目的实际影响

**几乎为零**。智能照明这种控制类 MLP 模型对精度非常鲁棒——亮度 80.5% 还是 81.2%，人眼根本看不出区别。量化带来的精度损失在本场景中完全可以忽略。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    量化风险评估                                           │
├──────────────────┬────────────────────┬──────────────────────────────────┤
│ 模型类型          │ 量化风险            │ 原因                             │
├──────────────────┼────────────────────┼──────────────────────────────────┤
│ 医疗影像 CNN      │ 高（可能不可接受）   │ 细微特征差异影响诊断结果            │
│ 语音识别 RNN      │ 中（需评估）        │ 频谱细节丢失影响识别率              │
│ 智能照明 MLP ✓   │ 极低（可忽略）       │ 输出为连续控制量，人眼无法区分       │
└──────────────────┴────────────────────┴──────────────────────────────────┘
```

#### 量化方式选择

| 量化方式 | 说明 | 适用场景 |
|----------|------|----------|
| 训练后量化 (Post-Training Quantization) | 训练完成后直接转换，简单快捷 | 本项目 MVP 推荐 |
| 量化感知训练 (Quantization-Aware Training) | 训练时就模拟量化误差，精度更高 | 精度敏感场景 |

**MVP 推荐使用训练后量化**，流程简单且对本项目精度影响可忽略。

## 结论与行动项

### 四个议题的收敛结论

| 议题 | 核心结论 | MVP 行动 |
|------|----------|----------|
| 模型选型 | 使用 MLP（全连接网络），不涉及 CNN/Transformer | 设计极简 MLP，目标 < 5KB |
| 数据策略 | 训练数据与推理数据天然一致，用合成数据快速启动 | 编写 Python 脚本生成合成数据集 |
| 概念漂移 | 通过提升泛化能力覆盖 80%+ 场景，MVP 不做 OTA/影子模式 | 在合成数据中加入极端场景 |
| 软硬协同 | TDD 驱动，从 PC 单元测试到真机实测逐步验证 | Twister + native_sim → ESP32-S3 烧录验证 |
| 网络结构 | MLP 处理一维数值，CNN 处理图像，Transformer 处理序列；本项目只需 MLP | — |
| 推理本质 | AI 推理是函数拟合（权重公式），不是查表或 if-else；体积固定、自动内插 | — |
| 泛化策略 | 质量 > 数量，多样性 > 重复性；80% 高质量场景数据可拟合 99% 体验 | 合成数据需均衡覆盖各边界条件 |
| 模型量化 | INT8 量化后体积缩为 1/4，本项目精度损失可忽略；推荐训练后量化 | 使用训练后量化导出 INT8 .tflite |

### MVP 开发优先级

1. **编写数据生成脚本**：用 Python 生成合成训练数据（含极端场景）
2. **训练 MLP 模型**：极简结构，导出 `.tflite`
3. **PC 端 TDD 验证**：Twister 单元测试确保 TFLM API 调用正确
4. **真机实测**：烧录到 ESP32-S3，验证推理耗时和内存占用
5. **迭代优化**：根据实测结果调整模型结构和数据
