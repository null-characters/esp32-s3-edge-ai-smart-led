# 架构巨变：从"感知"跨越到"交互"

> **核心命题**：如何让一个智能照明网关从"默默观察的眼睛"，变成一个"听得懂人话的大脑"？

---

## 一、架构演进的本质

## 二、从"眼睛"到"大脑"的跨越

### 2.1 感知设备的局限

| 维度 | 感知设备 | 问题 |
|------|----------|------|
| **信息流** | 单向 | 用户无法输入意图 |
| **决策权** | AI 独占 | 用户无法覆盖决策 |
| **可解释性** | 黑盒 | 用户不知道为什么这样调光 |
| **个性化** | 无 | 无法表达"我想要..." |
| **容错性** | 低 | AI 判断错误，用户无法纠正 |

**典型场景痛点**：

```
场景：AI 判断"会议高潮"，自动调到 6500K 100% 亮度
用户真实意图：正在进行投影演示，需要灯光调暗
结果：用户只能手动关灯，体验极差
根本原因：AI 无法理解用户的真实意图
```

### 2.2 交互设备的优势

| 维度 | 交互设备 | 优势 |
|------|----------|------|
| **信息流** | 双向 | 用户可输入意图 |
| **决策权** | 人机协作 | 用户可覆盖 AI 决策 |
| **可解释性** | 透明 | 用户问"为什么这样调？"，AI 可解释 |
| **个性化** | 支持 | 用户说"我想要暖光"，AI 记忆偏好 |
| **容错性** | 高 | AI 错误，用户语音纠正 |

**同样的场景**：

```
场景：AI 判断"会议高潮"，自动调到 6500K 100% 亮度
用户语音："小白网关，切换到演示模式"
AI 响应：立即切换到投影演示模式（主灯 30%，黑板灯 100%）
结果：用户满意，体验流畅
根本原因：AI 听懂了用户的真实意图
```

## 三、双轨并行架构设计

### 3.1 架构理念：人脑的双系统理论

借鉴认知心理学中**卡尼曼的"快思考与慢思考"理论**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    人脑双系统 vs 设备双轨架构                              │
├──────────────────┬────────────────────┬──────────────────────────────────┤
│ 人脑系统         │ 特征                │ 对应设备轨道                      │
├──────────────────┼────────────────────┼──────────────────────────────────┤
│ 系统1：快思考    │ 直觉、自动、无意识   │ 轨道一：自动感知控制               │
│                  │ 毫秒级响应          │ 雷达+麦克风 → AI → 自动调光        │
│                  │                     │ 后台持续运行                       │
├──────────────────┼────────────────────┼──────────────────────────────────┤
│ 系统2：慢思考    │ 理性、控制、有意识   │ 轨道二：语音命令控制               │
│                  │ 秒级响应            │ 用户语音 → ESP-SR → 执行命令       │
│                  │                     │ 唤醒词触发后激活                   │
└──────────────────┴────────────────────┴──────────────────────────────────┘
```

| 场景 | 雷达特征 | 声音特征 | 自动调光 |

**支持的语音命令**：

| 命令类型 | 示例 | 效果 |
|----------|------|------|
| **场景切换** | "打开专注模式" | 切换到专注场景（4000K, 60%） |
| **亮度调节** | "调亮一点" | 亮度 +20% |
| **色温调节** | "换成暖光" | 色温切换到 3000K |
| **模式切换** | "切换演示模式" | 主灯 30%, 黑板灯亮 |
| **查询状态** | "现在是什么模式" | TTS 回复（需DAC+扬声器） |
| **取消操作** | "恢复自动" | 回到轨道一自动控制 |

> **⚠️ 硬件闭环警告**：TTS 语音回复需要 I2S DAC（如 MAX98357A）和扬声器。
> 当设备自身发声时，麦克风会录入设备播放的声音，导致：
> - 无限循环（设备听到自己的声音再次触发）
> - 误识别（将 TTS 内容识别为命令）
>
> **解决方案**：必须在 ESP-ADF 中配置 **AEC（回声消除）** 流水线，
> AEC 需要硬件提供一路 Reference（参考音频）信号。

```
│                                │                                        │
│                                ▼                                        │
│                        ┌───────────────┐                               │
│                        │  执行语音命令  │                               │

│              │                 │                 │                     │
│        TTL过期          环境状态变化        用户说"恢复自动"            │

└─────────────────────────────────────────────────────────────────────────┘
```

**仲裁规则**：

| 规则 | 说明 |
|------|------|
| **语音优先** | 用户语音命令覆盖自动决策 |
| **TTL租约** | 语音干预带有效期（默认2小时），过期自动释放 |
| **环境退出** | 雷达检测环境重大变化（人离开>10分钟），自动释放语音锁 |
| **冲突处理** | 用户说"关灯"，雷达检测有人 → 听用户的 |
| **恢复机制** | TTL过期/环境变化/用户说"恢复自动" → 回到轨道一 |

## 四、技术实现路径

### 4.1 技术栈演进

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    技术栈演进对比                                          │
├──────────────────┬────────────────────┬──────────────────────────────────┤
│ 维度              │ 当前（Zephyr）      │ 目标（ESP-IDF）                   │
├──────────────────┼────────────────────┼──────────────────────────────────┤
│ 操作系统          │ Zephyr RTOS        │ ESP-IDF (FreeRTOS)               │
│ 语音识别          │ 无                 │ ESP-SR                           │
│ 音频框架          │ 自实现 I2S         │ ESP-ADF + AEC 流水线             │
│ 多模态推理        │ TFLM               │ TFLM (ESP-IDF 版本)               │
│ 唤醒词            │ 无                 │ WakeNet                          │
│ 命令词            │ 无                 │ MultiNet                         │
│ 音频前端          │ 无                 │ AFE (AEC/BSS/NS)                  │
│ 音频输出          │ 无                 │ I2S DAC (MAX98357A) + 扬声器      │
└──────────────────┴────────────────────┴──────────────────────────────────┘
```

### 4.2 ESP-IDF 生态集成

| 组件 | 功能 | 集成难度 |
|------|------|----------|
| **ESP-SR** | 语音识别框架 | 低（官方示例丰富） |
| **ESP-ADF** | 音频开发框架 | 低（组件化设计） |
| **ESP-Skainet** | 智能语音助手 | 低（完整方案） |
| **TFLM** | 多模态推理 | 中（需重新集成） |

### 4.3 迁移工作量

| 模块 | 工作量 | 备注 |
|------|--------|------|
| 基础框架迁移 | 1 周 | LED/Wi-Fi/UART/I2S |
| 多模态推理迁移 | 1 周 | TFLM 集成 |
| ESP-SR 集成 | 1 周 | 唤醒词 + 命令词 |
| 双轨架构整合 | 1 周 | 优先级仲裁 |
| **总计** | **3-4 周** | - |
